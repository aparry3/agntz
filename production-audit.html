agntz Production-Readiness Audit

2026-05-20 · branch feat/playground · scope: worker, app, core, store-postgres

Executive Summary

Functionally complete, but not safe to launch as a hosted multi-tenant service in current state. Architecture is sound — clean provider abstraction, real telemetry primitives, transactional Postgres store, AI-SDK-based runner. But there are 6 verified critical gaps an attacker or a single bad-actor tenant could exploit on day one, plus a handful of scalability gaps that will hurt somewhere between 100 and 10,000 requests/day.

This audit covered six dimensions in parallel: worker scalability, database, security, observability & reliability, deployment & ops, and AI/LLM integration. The four most consequential claims (CORS, SIGTERM, internal-auth, HTTP-tool SSRF) were verified directly against source.

Critical — Fix before any public hosted launch

These aren't "production hardening." These are "someone will find this in week one."

1.HTTP-tool SSRF: no host validation

Critical Verified

packages/core/src/http-tool.ts:331 calls fetch(finalUrl, ...) with zero validation of the target.

Meanwhile packages/core/src/image-fetcher.ts:129-207 has a complete SSRF defense — rejects file://, localhost, 169.254.0.0/16 (AWS metadata), RFC1918, IPv6 loopback/ULA, IPv4-mapped IPv6. The defense was written. It just isn't reused for the HTTP tool.

What this means: a user who can save a manifest can define url: "http://169.254.169.254/latest/meta-data/iam/security-credentials/", run the agent, and exfiltrate the worker's IAM role credentials. Or probe http://localhost:5432, or internal Railway services.

Fix
Extract the validator from image-fetcher.ts into a shared assertSafePublicUrl(). Apply in http-tool.ts before fetch(), in webhook delivery, and in validateManifest() so SSRF URLs fail at save-time, not just run-time.

2.CORS wide open

Critical Verified

packages/worker/src/routes.ts:106:

app.use("*", cors());

No config. Hono's default is permissive: Access-Control-Allow-Origin: * (or echoes the request origin) and methods GET, HEAD, PUT, POST, DELETE, PATCH.

Less bad than it looks for API-key auth (browsers don't send Authorization cross-origin with *) — but still wrong, and becomes critical the moment any browser-targeted auth (cookies, custom headers) lands on the worker.

Fix
Lock to known origins: cors({ origin: [process.env.APP_URL, process.env.SITE_URL], credentials: true }).

3.Internal-secret auth has all-tenant blast radius

Critical Verified

packages/worker/src/middleware/auth.ts:25-39: if the request carries X-Internal-Secret, the worker takes the userId from the request body or X-User-Id header. The secret is a static env var loaded once on boot. No signing, no per-user binding, no rotation hook.

If WORKER_INTERNAL_SECRET leaks (logs, env-var paste, ex-employee, supply chain), the attacker becomes any user by passing their userId in the body. No audit log of this happening.

Fix — minimum
HMAC-sign the (userId, timestamp, body-hash) tuple with the shared secret instead of using the secret as a static bearer. Reject signatures > 60s old.
Fix — better
Generate a per-request JWT in the app that embeds the Clerk-verified userId, signed with a key the worker can verify.

4.No rate limiting, no cost cap

Critical

No per-tenant rate limit on /run, /run/stream, or /runs. No per-tenant token-spend cap. Token budget in core/src/runner.ts:1629 is per-invocation — a tenant can launch invocations in a loop and the Anthropic bill is the only stop sign.

model-pricing.ts computes USD per call and writes it to a span attribute, but it's never persisted to InvocationLog, never aggregated per tenant, and never checked against a quota.

Fix
Minimum: in-process token-bucket per userId on the run endpoints, plus a daily-spend table the runner consults before each LLM call. Long-term: Redis or Postgres so it survives replica restart.

5.No SIGTERM / graceful shutdown

Critical Verified

packages/worker/src/server.ts:22-26: serve() is called and the file ends. No process.on('SIGTERM', ...). When Railway redeploys, the container is killed mid-stream: in-flight LLM calls leak, pending trace spans buffered in trace-registry.ts (flushes every 250ms / 100 spans) are lost, SSE clients see TCP RST with no terminal event, Run records remain stuck in running state until eviction.

trace-registry.ts:189-204 already has waitForFlush(). It's just never called.

Fix
On SIGTERM: stop accepting new requests, signal AbortController for all in-flight runs (30s deadline), await traceRegistry.waitForFlush() and runRegistry.drain(), then exit.

6.In-memory registries block horizontal scaling

Critical

worker/src/trace-registry.ts:44-242 (InMemoryTraceRegistry) and core/src/run-registry.ts:70-107 (InMemoryRunRegistry) hold live SSE subscribers and in-flight run state in per-process maps. A second worker replica behind a Railway load balancer cannot serve the same trace stream as replica #1 — the client reconnects to a replica with no knowledge of the trace.

Fallback at routes.ts:603-651,770-807 does a one-shot snapshot from the store after eviction, but it's not live — real-time updates lost on replica failover.

Fix — cheap
Force sticky sessions at the load balancer (Railway supports this).
Fix — correct
Swap both registries for Redis-backed implementations using pub/sub for fanout. Both already sit behind interfaces, so the swap is contained.

High — Will hurt at thousands/day or before

7.No structured logging, no request/trace correlation

High

26+ console.log() calls in worker/src/routes.ts alone. Plain text, no JSON, no traceId threaded in. When something breaks at 3am you cannot grep Railway logs by tenant or by run.

Fix
Swap console.* for a pino instance (~50 lines). Thread traceId/runId/userId through via Hono's c.set().

8.No error tracking

High

Zero references to Sentry/Honeybadger/Rollbar across the repo. try/catch blocks log to console and return {error: errorMessage(err)} — stack trace dropped at core/src/runner.ts:337: err instanceof Error ? err.message : err.

Fix
Drop in @sentry/node in worker/src/server.ts and app/src/middleware.ts. Capture exceptions with userId, runId, traceId tags.

9.Health check is a stub

High Verified

worker/src/routes.ts:108-110:

app.get("/health", (c) => {
  return c.json({ status: "ok", service: "agntz-worker" });
});

Returns 200 unconditionally. Does not check Postgres connectivity, LLM provider reachability, or memory pressure. Railway will route traffic to a broken worker because the health probe lies.

Fix
/health does SELECT 1 with 1s timeout. /health/deep additionally pings the configured LLM provider.

10.No circuit breaker on LLM providers

High

core/src/utils/retry.ts does 3-attempt exponential backoff (good). But if Anthropic is degraded for an hour, every request retries 3× with 1-30s delays, piling Promises onto the event loop. No fast-fail.

Fix
Wrap the model call with a sliding-window failure counter (opossum-style). After N failures in M seconds, return 503 immediately for the next D seconds.

11.Missing composite indexes

High

store-postgres/src/postgres-store.ts has single-column indexes on most lookup paths, but lacks composites for the actual query shapes:

  • listTraces filters by (owner_id, agent_id, status, started_at) — current index is just (owner_id, started_at).
  • listRuns filters by (user_id, parent_id, agent_id, status, started_at) — current index is (user_id, started_at).

At 1000 traces/day per user these still scan acceptably. At 100k they don't.

Fix
Add the two composite indexes; ship a migration.

12.No retention policy / trace TTL

High

store.deleteOlderThan() exists in postgres-store.ts:1456-1485 but is never called. No cron. Trace data grows linearly forever. At ~10MB/day for modest traffic, you cross 100GB inside a year.

Fix
Nightly job (cron container, or setInterval in the worker behind a leader-election flag) that calls deleteOlderThan(cutoff) per tenant. Free-tier 30-day retention, paid configurable.

13.No idempotency keys

High

POST /run and /runs have no Idempotency-Key handling. A client that retries on network blip creates two runs and double-charges the LLM bill.

Fix
Accept Idempotency-Key header. Store (userId, idempotencyKey) → runId in Postgres with 24h TTL. Return existing run on duplicate.

14.Long requests block load-balancer timeouts

High

/run and /run/stream block the HTTP connection for the full agent duration (routes.ts:226,369). Many proxies/CDNs idle-timeout at 60-100s. Anything > 1 min gets cut mid-stream from the client's perspective, even though the worker keeps running.

Fix
Push users toward the async /runs endpoint (it exists) + /runs/:id/stream for results. Document /run/stream as best for short interactive runs only.

15.Webhook URLs not SSRF-validated

High

Same class of bug as #1, in core/src/webhooks/dispatcher.ts. A user can register a webhook to http://internal-jenkins/build/... and trigger it.

Fix
Apply the shared assertSafePublicUrl() at webhook creation and again at delivery.

16.Manifest validation doesn't catch SSRF URLs at save-time

High

packages/manifest/src/validate.ts:266-298 validates HTTP tool URLs as syntactically valid, but doesn't run the SSRF check. A tenant can save a malicious manifest, sit on it, then either run it themselves or (if multi-tenant ever shares agent libraries) get someone else to.

Fix
Call assertSafePublicUrl() from manifest validation. Save-time rejection > run-time rejection.

17.PII in logs / no redaction layer

High

Prompts and LLM responses are stored in InvocationLog / sessions / traces with no PII redaction. The HTTP-tool auth redaction (core/src/auth/redact.ts) only scrubs known auth-token patterns from response bodies, not arbitrary PII.

If a prompt includes "customer record: {name, ssn, email}", it's persisted indefinitely. GDPR/CCPA delete-on-request has no documented path.

Fix
Add PII redaction in prompts before storage (mask emails, SSNs, credit cards via regex). Document a per-user purge endpoint. Add retention policy that complies with default expectations (90d?).

18.MCP server trust model

High

core/src/mcp/client-manager.ts:108-175 connects to user-supplied MCP servers and trusts the returned tool definitions wholesale. If an MCP server is compromised or user-controlled, it can inject malicious tool definitions; tool execution is not sandboxed.

Resources returned by MCP servers can also become prompt-injection vectors if embedded in conversation history.

Fix
Document MCP servers as a code-execution trust surface. Allowlist known MCP servers for the hosted offering, or scope user-supplied MCP servers strictly to that user (don't share results across tenants).

19.Tool whitelist not enforced at runtime

High

Manifest's tools: array is a declaration, not a runtime gate. If a tool is registered in the process-wide ToolRegistry (core/src/tool.ts:110-122) the LLM can call it whether the manifest listed it or not.

Combined with core/src/runner.ts spawn-agent functionality, no documented depth limit on recursive agent spawning.

Fix
Before executing a tool, check the call is in the manifest's declared tools. Add recursion depth limit to spawn-agent (already has DEFAULT_MAX_RECURSION_DEPTH = 3 — verify it's reachable from spawned children).

Medium — Becomes critical past 10k/day

20.pg.Pool with no explicit sizing

Medium

postgres-store.ts:415 instantiates new Pool({ connectionString }) with default max: 10. Worker + app both instantiate stores, doubling demand. No connection timeout, no queue bound.

Fix
new Pool({ ...opts, max: 20, idleTimeoutMillis: 30000, connectionTimeoutMillis: 2000 }).

21.No HTTP keep-alive pool for tool calls

Medium

core/src/http-tool.ts:331 uses default fetch — fresh TCP+TLS handshake per tool call. File-descriptor and handshake overhead grows with concurrency.

Fix
Use undici.Agent with keep-alive and bounded socket pool. Pass via dispatcher option.

22.Serial tool execution

Medium

core/src/runner.ts:1733: for (const tc of toolCalls) await executeToolCall(tc). Each tool blocks the next. 10 tool calls of 1s each = 10s instead of ~1s.

Fix
Promise.all(toolCalls.map(executeToolCall)) + mutex on shared run state. Tool side effects must be commutative or explicitly serialized.

23.MCP client.callTool() has no timeout

Medium

core/src/mcp/client-manager.ts:148-164 awaits callTool() with no per-call timeout. A slow MCP server stalls the agent loop past the agent-level timeout.

Fix
Wrap with AbortSignal.timeout(remainingAgentBudget); fallback to error message returned to LLM.

24.PgBouncer incompatibility

Medium

postgres-store.ts:477 uses pg_advisory_lock() in the migration runner. Advisory locks are session-scoped; under PgBouncer transaction-mode pooling the lock is lost between statements.

Fix
Use a dedicated migration lock table (INSERT ... ON CONFLICT DO NOTHING for acquisition) if/when PgBouncer goes in front of Postgres.

25.No CI deploy automation

Medium

Every deploy is a manual Railway/Vercel click. One wrong click ships a broken worker. CI runs tests + lint on PR but doesn't deploy.

Fix
Auto-deploy main to staging via Railway's GitHub action. Manual promotion to prod via tag push (release workflow already exists for npm).

26.One-way destructive migrations

Medium

Migrations in postgres-store.ts include DROP TABLE (v3 line 142), DROP TABLE IF EXISTS ar_workspaces CASCADE (v4 line 154), and TRUNCATE (v4 line 171). Rolling back a code deploy after one of these = data loss + crash on missing tables.

Fix
Adopt expand/contract migration pattern: add new columns/tables, deploy code that writes to both, deploy code that reads from new only, then drop old in a separate later migration. Never destructive in a single step.

27.Single-region deployment

Medium

Railway worker deploys to one region (US-East by default). Global users face 200-300ms latency just for the request roundtrip, on top of LLM call time.

Fix
Not urgent. When it matters: replicate worker to EU + APAC regions, point Cloudflare Workers as edge proxy to route by geography. Postgres stays single-primary; add read replicas if needed.

28.Container base images unpinned

Medium

Dockerfile.worker:4, Dockerfile.app:4, Dockerfile.site:4 all use node:22-slim without sha256 digest. Resolves to latest tag at pull time. No USER directive — containers run as root.

Fix
Pin to node:22.11.0-slim@sha256:<digest>. Add RUN groupadd -r app && useradd -r -g app app and USER app before CMD.

29.No resource limits configured

Medium

railway.json has no memory or CPU configuration. Railway defaults to 512MB which can OOM on large multimodal invocations.

Fix
Add "resources": { "memoryMb": 2048, "cpuUnits": 2000 } to railway.json. Watch p99 memory in prod and tune up.

30.Some list endpoints unpaginated

Medium

listConnections (postgres-store.ts:1046), listSkills (:1501), listProviders (:994) have no LIMIT. listRuns and listTraces do (good).

Fix
Add LIMIT 200 default + cursor pagination to all list methods.

31.Trace span loss on crash

Medium

worker/src/trace-registry.ts:233 catches DB write errors and silently swallows them. Spans in the pending buffer (up to 100 spans / 250ms) are lost on crash. insertSpansBatch().catch(() => {}) treats all errors as acceptable loss.

Fix
Outbox pattern: write spans to a ar_span_outbox table inside the run's transaction. Async worker reads outbox → ar_spans. Crash-safe.

32.Cost computed but not persisted or enforced

Medium

core/src/runner.ts:1668 calls computeCost() after each model call. Sets it on the telemetry span. Not stored in InvocationLog. Not checked against per-tenant budgets. DEFAULT_RATES in model-pricing.ts:15-22 is hardcoded to 2026-05 prices.

Overlaps with #4 (rate limiting) — they're the same family of "no economic guardrails."

Fix
Persist cost_usd on InvocationLog. Tenant-level rolling total in Postgres. Pre-call budget check in the runner. Load rates from config table, not hardcoded.

33.Default maxSteps=10 may be too tight

Medium

core/src/runner.ts:107: DEFAULT_MAX_STEPS = 10. One LLM call → 3 tool calls → 3 more = 9 steps. Agents that legitimately need multi-stage planning fail silently.

Fix
Bump default to 20. Document per-agent override in manifest.

34.ToolRegistry has no concurrent-write guard

Medium

Global ToolRegistry (core/src/tool.ts:39-54) has no lock. Two concurrent agents loading the same skill both check .has(), both see false, both register. Last write wins.

Not a problem with current Node.js single-thread model, becomes a problem the moment worker threads or per-tenant subprocesses land.

Fix
Make register() atomic check-then-set. Acceptable to keep single-threaded assumption explicit in a comment if you don't plan to thread.

35.HTTP-tool timeout fixed at 30s, ignores agent budget

Medium

core/src/http-tool.ts:335: AbortSignal.timeout(30_000) baked in. If agent has timeoutMs: 60_000 and the first tool takes 30s, the agent has only 30s left for everything else — but the HTTP tool will still spend another 30s on the next call.

Fix
Compute remainingMs = Math.max(agentTimeoutMs - elapsedMs, 5000) and use that as the HTTP tool timeout.

What it takes to handle thousands of requests/day

The blunt answer: at thousands of requests/day (~0.05-0.5 RPS sustained, bursts to maybe 10 concurrent runs), one well-tuned worker replica handles it on a single Railway service with no architectural change. The bottleneck is not throughput — it's the failure modes. The 6 critical blockers above all bite at low traffic if you're unlucky (one SSRF run, one credential leak, one SIGTERM mid-stream).

The question you actually asked: built for scale, not scrambling at scale. So the architectural moves, in priority order:

  1. Make the worker truly stateless. Replace InMemoryTraceRegistry and InMemoryRunRegistry with Redis-backed implementations using Redis Streams or pub/sub for SSE fanout. Both already sit behind interfaces — contained swap, ~2-3 days work plus testing.
  2. Move long runs off the request path. /runs already supports async start. Lean into it: run-create returns immediately with a runId; execution happens in a worker-pool process (separate Railway service, or BullMQ / pg-boss queue). HTTP-server replicas only handle short-lived API calls and SSE fanout from Redis. Decouples "scale request handling" from "scale agent execution" — very different CPU/memory profiles.
  3. Persist what matters synchronously, batch what doesn't. Run state and final results go to Postgres in the request path. Trace spans batch (already do) — but back them with a durable queue (Redis list or Postgres outbox table) so a crashed worker doesn't lose them.
  4. Quotas + circuit breakers everywhere. Per-tenant rate limit on /run* endpoints (Redis token bucket). Per-tenant daily LLM spend cap, checked before each model call. Provider-level circuit breaker that fails fast when Anthropic is down. The difference between "one bad tenant degrades everyone" and "one bad tenant gets 429'd, everyone else is fine."
  5. Observability that survives replicas. Structured JSON logs with traceId/userId/runId. Sentry for errors. A real /health that probes its dependencies. OpenTelemetry export from telemetry.ts to a backend you can actually query (Honeycomb, Datadog, Grafana Cloud).
  6. DB hygiene. Composite indexes. Retention cron. PgBouncer in front of Postgres (after fixing the advisory-lock migration). Read replicas if/when listing endpoints get hot — they won't, at thousands/day.
