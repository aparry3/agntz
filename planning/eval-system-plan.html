Agntz planning Eval system

First-class agent evals

A platform plan for reusable datasets, rubric-driven scoring, version comparisons, and an Agntz-owned evaluator agent with structured output.

Summary Core model Scoring contract APIs, SDK, CLI, UI Execution design Test plan Assumptions

Summary

Build evals as first-class workspace records stored separately from agent manifests. Each eval is linked to one agent, can use reusable workspace datasets, runs asynchronously, snapshots its inputs at run time, and stores full per-case scoring detail for history and comparison.

Separate records Keep manifests clean

Agent YAML remains focused on behavior. Evals, datasets, and results live in their own store interfaces and app routes.

Reusable data Workspace datasets

Datasets are reusable assets that can be attached to evals and used for repeated version comparisons.

Version-aware Snapshot on run

Every eval run stores immutable snapshots of the eval, dataset, and target agent version used to produce the score.

Core model

Add a core eval domain model and a new EvalStore that is implemented by memory, SQLite, and Postgres stores.

Entity Purpose Important fields
EvalDefinition Rubric and settings for evaluating one agent. id, agentId, name, criteria[], defaultDatasetId?, passThreshold, judgeModel?
EvalCriterion One weighted scoring dimension. id, name, description, weight = 1, threshold?
EvalDataset Reusable workspace-level cases. id, name, items[], metadata?
EvalDatasetItem One input and optional reference output. id, input, expected?, metadata?
EvalRun Async execution record and immutable result. id, status, snapshots, caseResults, summary
The previous low-level AgentDefinition.eval support is removed. The productized system uses separate eval records rather than adding eval YAML to manifests in v1.

Scoring contract

Use a hidden Agntz system judge agent to evaluate outputs against rubric criteria. The judge returns a stable envelope with dynamic criterion results keyed by stable criterion ids. 

{
  "overallScore": 0.82,
  "passed": true,
  "criteria": {
    "accuracy": {
      "score": 0.9,
      "passed": true,
      "reason": "The answer matches the expected refund window."
    },
    "tone": {
      "score": 0.75,
      "passed": true,
      "reason": "Polite and concise, but slightly generic."
    }
  },
  "reason": "Strong answer with minor tone issues."
}
  • Criterion id is the machine key; criterion name is display-only.
  • overallScore is a weighted average: sum(score * weight) / sum(weight).
  • Every criterion defaults to weight: 1, so the default is a plain average.
  • Each criterion and eval run stores both score and passed.
  • The eval run passes when overallScore >= passThreshold; default threshold is 0.7.

APIs, SDK, CLI, UI

Hosted API

  • GET/POST /api/evals
  • GET/PUT/DELETE /api/evals/:evalId
  • GET/POST /api/datasets
  • GET/PUT/DELETE /api/datasets/:datasetId
  • POST /api/eval-runs to start an async run.
  • GET /api/eval-runs/:runId to retrieve status, summary, snapshots, and case results.
  • GET /api/eval-runs?agentId=&evalId=&datasetId= for history.

Clients and CLI

  • Add client.evals.list/create/get/update/delete/run/getRun/listRuns.
  • Add client.datasets.list/create/get/update/delete.
  • Add local SDK support via agntz({ agents, evals?, datasets? }).
  • Expose client.evals.run({ agentId, evalId, datasetId, version? }).
  • Add CLI commands: agntz eval run, agntz eval runs, and agntz eval get.

App UI

  • Add an Evals mode inside the individual agent view.
  • Show eval definitions, attached/default dataset, latest score, run history, and version comparison.
  • Run detail shows input, expected output, agent output, criterion scores/reasons, pass/fail, and linked agent run ids.
  • V1 comparisons are scoped to versions of the same agent: current, latest, aliases, or timestamped versions.

Execution design

  • POST /api/eval-runs creates an EvalRun immediately with status: "running".
  • The executor resolves the target agent version, eval definition, dataset, and system judge configuration.
  • Each dataset item invokes the target agent with item.input.
  • The judge receives input, expected output, actual output, metadata, and criteria.
  • The judge response is validated against the generated structured output schema.
  • Each case stores output, usage, duration, linked run id, per-criterion results, and aggregate score.
  • On judge parse/model failure, mark that case failed with score 0, preserve the reason, and continue.
  • Cancellation stops scheduling new cases and marks remaining cases skipped or cancelled.

Test plan

  • Core tests for weighted scoring, threshold semantics, criterion ids, judge output validation, snapshots, and agent version resolution.
  • Store conformance tests for eval, dataset, and eval run CRUD with user scoping and filters.
  • API/client tests for hosted CRUD, async run lifecycle, malformed requests, and missing agent/eval/dataset handling.
  • SDK and CLI tests for local file loading, local eval execution, JSON output, and remote API delegation.
  • UI tests for creating/editing evals, selecting datasets, starting runs, inspecting case results, and comparing versions.

Assumptions

  • V1 eval definitions are linked to one agent.
  • Datasets are reusable workspace assets, not owned by one agent.
  • V1 comparison supports versions of the same agent only, not arbitrary agent bakeoffs.
  • V1 stores full case detail by default: input, expected, output, criterion scores, reasons, and linked run references.
  • V1 uses snapshot-on-run instead of explicit eval/dataset version records.
  • The system judge agent is hidden and Agntz-owned, with a default judge model unless an eval specifies judgeModel.
