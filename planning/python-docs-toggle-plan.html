Python and TypeScript docs toggle for Agntz

The Python SDK is now on the branch, but the public site still reads as TypeScript-first. This plan adds a language toggle to docs and landing snippets so Python users can follow the same path from install to agent execution without translating TypeScript examples by hand.

Design principles

One concept path

Users should learn the same Agntz concepts regardless of language: YAML manifests, local SDK, hosted client, sessions, runs, traces, tools, and deployment.

Shared YAML stays shared

Agent definitions are the portable contract. YAML snippets should not be duplicated unless the feature itself differs by runtime.

Language-specific code toggles

SDK snippets should switch between TypeScript and Python at the same point in the reading flow, similar to common docs systems like LangChain.

No false parity claims

The docs should clearly mark first-slice gaps, such as local Python token streaming and the eval product not being reimplemented yet.

Implementation architecture

Layer 1

Shared language preference

Add a docs/site language state with "ts" and "python". Persist it to localStorage, optionally hydrate from ?lang=python, and expose a compact segmented control at the top of docs pages.

Layer 2

Grouped code fences in docs markdown

Extend packages/site/src/components/docs/markdown.tsx so paired fences render as one language-aware snippet. 

```ts [index.ts] {group=quickstart-run}
...
```

```python [main.py] {group=quickstart-run}
...
```

Ungrouped YAML, bash, HTTP, and text snippets remain always visible. Raw markdown and /llms.txt should still include both language examples.

Layer 3

Code block language support

Add Python highlighting to packages/site/src/components/landing/code-block.tsx and let docs code blocks pass lang="python". Keep copy behavior scoped to the visible snippet in rendered docs.

Layer 4

Landing snippet toggle

Reuse the same language state or a small matching hook for the landing page. Hero snippets, install commands, and CTA snippets should switch between TypeScript and Python.

Landing page snippet updates

Hero

  • Keep agent.yaml as the shared first tab.
  • Toggle runner.ts and runner.py.
  • Toggle install pill between npm install @agntz/sdk and pip install "agntz[litellm]".

The Shift

  • Keep the declarative YAML example shared.
  • Toggle the final call snippet between client.agents.run({...}) and client.agents.run(...).
  • Do not make the comparison feel like a language war. The point is runtime vs hand-wired loop.

Final CTA

  • Toggle install and run command lines.
  • Keep the CTA visually compact and first-action oriented.

Audit

  • Search all landing CodeBlock usages.
  • Add Python alternatives only where the snippet represents SDK usage.
  • Do not duplicate Docker, YAML, or conceptual examples unnecessarily.

Definition of done

A Python user can land on the site, switch to Python, install the package, load the same YAML, run an agent locally, use a hosted client, understand sessions/traces/tools, and see the current parity limits without leaving the docs flow.

Risks and mitigations

Parser complexity

Grouped markdown fences should stay simple. Avoid a general MDX-like parser. Use one explicit {group=name} convention and test it with the existing docs pages.

Stale snippets

Keep snippets aligned with the Python README and tests. Favor compact examples that map to covered API surfaces instead of aspirational samples.

Landing clutter

Do not add a second full product narrative for Python. The landing page should signal both SDKs while preserving the existing focused story: declarative YAML, runtime owns the loop.

Raw docs behavior

Rendered docs can hide one snippet at a time, but raw markdown and /llms.txt should preserve both language variants for AI readers and copy workflows.
