Proposed design · memrez

memrez: a memory layer for agntz

agntz can run agents, but it has no long-term memory — only verbatim sessions and uncurated context buckets. memrez is the durable layer: small tagged facts, partitioned by a hierarchical scope, curated over time. It is standalone, yet borrows agntz's own brain — its tagger and curator are agntz YAML manifests, run identically by the TypeScript and Python SDKs.

Problem & tenets

The closest existing primitives — SessionStore (verbatim history) and ContextStore (named scratch buckets) — are short-term and uncurated. memrez adds durable, deduped, queryable knowledge that accumulates across sessions.

Small entries + tags

Memory is many small facts, each tagged with topics — not one Markdown blob per topic. Tags are the index.

The agent navigates

A cheap scan returns a table of contents; the agent reads only the topics it actually needs, instead of loading everything.

Cheap writes, heavy curation

A fast model tags on write; a stronger model dedupes and reconciles contradictions in the background. The thinking happens in curate, off the hot path.

Tags-first

Deterministic scan→read over topics now. Embeddings and semantic search come later, as an additive strategy.

One brain, two runtimes

The tagger and curator are agntz YAML manifests — identical behavior in the TypeScript and Python SDKs, the same way agntz already keeps parity.

Concepts & data model

One memory is one entry, with two independent axes: a single owning scope (the silo) and many-to-many topics (the tags within it).

type Scope = string; // "global" | "org/acme" | "org/acme/user/u_123" | ".../session/s_abc"

interface MemoryEntry {
  id: string;                 // mem_<nanoid>
  scope: Scope;               // single owning scope
  content: string;            // one canonical fact ("Prefers metric units.")
  topics: string[];           // semantic tags within the scope
  type: "fact" | "preference" | "event" | "summary";
  source?: { agentId?: string; sessionId?: string; runId?: string };
  status: "active" | "superseded";
  supersededBy?: string;      // id of the entry that replaced this one
  createdAt: string;
  updatedAt: string;
  // embedding?: number[];    // Phase 6
}

interface TopicSummary {
  topic: string;
  count: number;               // live count of active entries
  blurb?: string;              // curate-maintained one-liner
  lastUpdatedAt: string;
  hasUncuratedWrites: boolean; // writes since the last curate
}

Scope resolution. read / scan over org/acme/user/u_123 see entries at that path and every ancestor prefix (org/acme, global), so org-wide and global facts surface for a user without being copied. Each entry owns exactly one scope; moving a fact up a level (session→user) is a curate operation, not multi-membership.

Public API

The four operations are plain methods (also what the tools and the cron job call):

import { createMemrez } from "memrez";

const memrez = createMemrez({
  store: sqliteStore("./memory.db"),       // default: in-memory
  reasoner: agntzReasoner({ client }),      // default impl wraps the agntz SDK + the two manifests
});

const toc     = await memrez.scan("org/acme/user/u_123");
const entries = await memrez.read("org/acme/user/u_123", "billing", { limit: 20 });
const result  = await memrez.write("org/acme/user/u_123", "Prefers email over phone.", {
  source: { sessionId, agentId },
});
const report  = await memrez.curate("org/acme/user/u_123");

Hosting is a one-line swap. The same interface — scan / read / write / curate / provider() / tools() — backed by a remote service instead of an in-process store + reasoner: 

import { createMemrezClient } from "memrez/client";

const memrez = createMemrezClient({ url: process.env.MEMREZ_URL, apiKey: process.env.MEMREZ_KEY });
// identical surface, now over HTTP — mem.tools() still returns kind: local handlers
scan(scope, opts?: { includeAncestors?: boolean; topicLimit?: number })
  : Promise<{ scope: Scope; topics: TopicSummary[] }>;

read(scope, topic, opts?: { limit?: number; includeAncestors?: boolean })
  : Promise<MemoryEntry[]>;

write(scope, content, opts?: { type?: EntryType; topicsHint?: string[]; source?: Source })
  : Promise<{ entry: MemoryEntry; action: "appended" | "superseded" | "deduped" }>;

curate(scope, opts?: { topics?: string[]; includeDescendants?: boolean })
  : Promise<CurateReport>;

Using it from an agntz agent

Memory is a first-class manifest capability, not a hand-wired tool — a memory: block sits beside agntz's existing skills and reply blocks. Declaring it makes the runner register memory_read / memory_write as system tools and, when autoScan is on, inject the scan TOC before the model call. 

# agents/support.yaml
memory:
  mode: read-write     # read-only | read-write (default); read-only registers only memory_read
  autoScan: true       # inject the scan TOC before the model call (the non-tool half)
  # no scope here — scope is per-run, from toolContext (never baked into the manifest)

You wire the implementation once, at app start, into the memory slot. mem is any MemoryProvider — embedded or a hosted client — and the YAML above never changes: 

const mem = createMemrez(options);          // or createMemrezClient({ url })

const client = await agntz({
  agents: "./agents",
  memory: mem.provider(),                   // the MemoryProvider slot
  tools: [...appTools],                     // memory_read / memory_write are NOT hand-listed
});
// load-time: an agent declaring `memory:` with no provider here → fail-fast, naming the agent

Locally, that's the whole setup: pnpm add memrez, construct a provider (in-memory store + a fake or real reasoner — no API key needed offline), inject it. agntz core never bundles memrez; it only defines the slot the provider fills.

Scope comes from toolContext, never the model. An agent cannot reach another user's memory by hallucinating a scope — it only picks the topic (read) or supplies the fact (write). The system tools read ctx.memrezScope; autoScan calls contextBlock(scope) with that same per-run scope.

Per-run scope still needs toolContext. The core createRunner().invoke(input, { toolContext }) threads it today (see the gymtext example), but @agntz/sdk's agents.run(...) does not yet — so the memory: wiring needs that passthrough added to the SDK (or the memory-bound agent runs on the core Runner). Tracked in open questions.

Scope as a capability

A scope is not just where a memory is filed — it is the access boundary. The isolation model is one rule:

The granted prefix is minted at the trust boundary from verified identity and is immutable for the request. The model can only refine downward within it — never widen, never jump sideways.

So scope has two halves: a granted prefix from auth (the API route, or a hosted workspace API key) that the model never controls, and an optional relative refinement supplied by the agent and joined under the prefix server-side — which can only narrow. 

effective_scope = granted_prefix + (optional refinement, clamped inside the prefix)

This is the discipline agntz already uses for ctx.user.id: the route loads identity and injects it via toolContext; the LLM chooses what to do, not whose data. A scope path is <tenancy…>/<subject> — leading segments are the namespace (app / workspace / org), the trailing segment is the subject (user / account). The exact shape is app-defined via a scope resolver, since only the app knows its tenancy model.

Inheritance goes up; isolation is sideways. A read at a scope includes its ancestors (broader, shared knowledge flows down) but never its siblings. That single rule delivers sharing and isolation together.

Example — gymtext (app → user)

gymtext                       ← trainer-wide memory (add later — costs nothing now)
└─ user/
   ├─ u_123   ← travel prefs, equipment access, goals, injuries
   └─ u_456   ← isolated: u_123's agent can never read this (sibling, not an ancestor)
// scope minted HERE, from the authenticated user — the trust boundary
const scope = `gymtext/user/${userId}`;
const toc = await memrez.contextBlock(scope);    // scan TOC as a string

const result = await runner.invoke("chat", `${toc}\n\n${message}`, {
  sessionId,
  toolContext: { user, memrezScope: scope },      // memory_read/write bind to this scope
});

memory_read / memory_write read ctx.memrezScope; the model picks a topic but the scope is fixed, so u_123 and u_456 can never see each other. Add trainer-wide memory later by writing at gymtext — every user's read inherits it, with no per-user duplication and no schema change. That free shared layer is why scope is a path, not a flat id.

Example — sales platform (app → org → account)

sales
└─ org/
   ├─ acme                    ← org heuristics: "if property A is true, also find B"
   │  └─ account/
   │     ├─ a_789  ← "usually omits property A; assume X, then confirm"
   │     └─ a_790
   └─ globex                  ← different org: fully isolated from acme

The lead agent for a_789 is granted sales/org/acme/account/a_789; its reads inherit sales/org/acme (the org's qualification rules) but globex is invisible. Promotion is the curator's job: running at sales/org/acme (granted the org subtree) it notices the same pattern across sibling accounts — "most omit property A" — and writes a generalized heuristic up at the org scope, where every account agent inherits it. Account quirks stay low; learned generalizations rise.

Running memrez: local vs hosted

"local" means two different things — separating them dissolves most of the confusion. Tool kind (kind: local) is how an agntz tool's handler is wired: an in-process function, versus kind: http (remote endpoint) or kind: mcp (MCP server). Topology is where memrez's store and reasoner physically live: embedded in your process, or a hosted service. They are independent — and the design pins the first and varies only the second.

The agent never changes; the provider does. You inject one MemoryProvider via the memory: slot. Whether mem is an in-process createMemrez(...) or a createMemrezClient({ url }) talking to a hosted service, the memory: block, the system tools, and your route code are identical — the same "graduate by changing one line" move agntz makes from @agntz/sdk to @agntz/client.

Why the system tools are local handlers. The runner-registered memory_read / memory_write run in-process before anything hits the wire: they bind scope from toolContext (the capability guarantee) and call the injected provider. A raw kind: http tool pointed straight at hosted memrez would instead hand request-shaping to the model and force scope through templated headers — more surface, weaker isolation. So the tools stay local; when the provider is a hosted client, that call is just a thin HTTP hop.

The scope model is identical in both — what changes is who is trusted to mint the prefix and where it is enforced. Path shape, inheritance, and isolation never change. gymtext (one app, its own DB) is naturally embedded; a multi-org sales SaaS is the case for hosted, because one process then holds many tenants' memories.

There is no separate "memrez agent server." The reasoner (tagger / curator) lives inside whatever mem you construct. Embedded, it runs in your process. Hosted, it runs inside the memrez service, which embeds the agntz SDK to execute the two manifests and owns the curate schedule. One service owns storage + reasoner + cron; you call it over HTTP.

The background curator is scope-bound too. A hosted deployment runs the curator as a scheduled per-scope job; the curator for org/acme is granted only the acme subtree and can never read another tenant. Every memrez operation — foreground tool call or background curate — runs inside a scope grant. A curator that could read across scopes would be a cross-tenant leak.

Reasoning layer (the shared brain)

Two agntz manifests, loaded by both runtimes from one canonical location (packages/memrez/agents/):

memrez-tagger · cheap model

In: { content, scope, existingTopics, topicsHint? }. Out: { topics, type, normalizedContent, duplicateOf? }. Drives write.

memrez-curator · stronger model

In: a scope's entry slice + the topic set. Out: { ops: CurateOp[] } — merge, supersede, retag, setBlurb, promoteScope. Drives curate.

memrez-core only knows the Tagger / Curator interfaces. agntzReasoner() is the default implementation that runs these manifests through client.agents.run(...); the per-language ModelProvider (AI SDK / LiteLLM) makes the actual LLM call. Swap in a custom reasoner and memrez has no agntz dependency at all.

Storage

MemoryStore is memrez's own small interface (mirrors how agntz ships MemoryStore / SQLiteStore):

putEntry · getEntry · supersede(ids, byId)
listTopics(scopePaths) → TopicSummary[]            // aggregates topic + count + blurb
getByTopic(scopePaths, topic, limit) → entries
getTopicMeta / setTopicMeta(scope, topic, blurb, lastCuratedAt)
listScopeSlice(scope, opts) → entries              // for curate

Tables: entries, topic_meta (scope, topic, blurb, lastCuratedAt). v1 adapters: in-memory + sqlite. Postgres + a vector column arrive with the embeddings phase.

scan TOC tuning (recommendation). Return all topic names + live counts (cheap even at hundreds of topics) plus curate-maintained blurbs; if topics exceed a cap, send blurbs only for the most-recently-touched K (names + counts for the rest). hasUncuratedWrites flags drift since the last curate. v1 is query-agnostic (the full map); query-aware ranking waits for embeddings.

Parity & contracts

Add contracts/memrez/ mirroring contracts/python-port/: scenario fixtures (a sequence of writes → expected topic assignment, supersede graph, scan TOC shape, scope-ancestor resolution). Both runtimes test against them.

Contract tests use a deterministic fake reasoner (a fixed content→topics mapping). That pins what must match across languages — storage, scope resolution, dedup, supersede, scan aggregation — without asserting on nondeterministic LLM output. Real tagger/curator quality is covered separately by agntz evals on the two manifests.

Implementation phases

Phase 1

memrez-core (TS), no LLM

Types, MemoryStore + in-memory adapter, scope/ancestor resolution, scan/read/write/curate orchestration against injectable Tagger / Curator. Tests with a fake reasoner. Cheapest way to find out if the scope/topic model feels right in real code.

Phase 2

agntz reasoner + manifests

memrez-tagger.yaml / memrez-curator.yaml; agntzReasoner() wiring write→tagger, curate→curator via the SDK. Evals on both manifests.

Phase 3

agntz manifest integration · agntz-core change

The memory: manifest block + MemoryProvider interface + validation (parse-time shape & reserved names; load-time fail-fast) in @agntz/manifest + @agntz/core. Runner wiring registers the system tools, honors mode, and injects the scan TOC on autoScan; mem.provider() fills the slot. One end-to-end example agent. Touches agntz core — mirror in Python (Phase 5), add manifest parity fixtures.

Phase 4

sqlite adapter

Persistence across restarts; topic_meta table.

Phase 5

Python port

Mirror core + adapters; load the same manifests via the Python agntz SDK; wire contracts/memrez/ into both test suites. Parity lock.

Phase 6

Embeddings · later

vector column (sqlite-vec / pgvector), embed on write + re-embed on curate, hybrid retrieval (topic filter + vector rank), query-aware scan.

Open questions

All have a recommended lean; none block Phase 1.

  1. Auto-injection. Read-side is resolved — the memory.autoScan flag makes TOC injection a declarative opt-in (the runner scans and prepends). Open sliver: should the runner ever auto-write (post-run fact extraction)? Lean: no — the caller passes finished facts (out of scope for v1).
  2. write dedup depth. How hard does the cheap tagger try to detect "already known" / supersede vs. always append and let curate clean up? Lean: append + exact-dup skip only; curate does the rest.
  3. Promotion trigger. session→user, or cross-sibling→parent (the sales "most accounts omit A" case). Curator-driven is the primary path; do we also expose an explicit promote() API? Lean: curator-driven, explicit API as an escape hatch.
  4. Blurb storage. topic_meta table vs a type:"summary" entry per topic. Lean: topic_meta.
  5. Package name. memrez is a separate package you inject as the provider — bare memrez (honors standalone) vs @agntz/memrez (monorepo consistency; signals the tight slot coupling). Doc assumes bare memrez.
  6. Where the slot lives. The memory: schema goes in @agntz/manifest; does the MemoryProvider interface live in @agntz/core (runner-adjacent, like SkillStore) or a thin shared contract package? Lean: alongside SkillStore in core.
  7. Manifest location across languages. Canonical YAML in packages/memrez/agents/; does Python read those files directly or get a packaged copy at build?
  8. SDK toolContext passthrough (agntz-side). The memory: wiring binds per-run scope from toolContext, which the core Runner's invoke(..., { toolContext }) threads today — but @agntz/sdk's agents.run() doesn't yet. Resolution: add toolContext to the SDK run options (small agntz-side change), or run memory-bound agents on the core Runner.

Out of scope (v1)

  • Embeddings / semantic search (Phase 6).
  • Extracting facts from raw transcripts (the caller's job for now).
  • Multi-scope-per-entry membership.
  • A hosted memrez service or UI.
  • Cross-tenant / global curate runs (curate operates on one scope subtree).
