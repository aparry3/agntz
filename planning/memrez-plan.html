memrez: a memory layer for agntz

agntz can run agents, but it has no long-term memory — only verbatim sessions and uncurated context buckets. memrez is the durable layer: small tagged facts, partitioned by a hierarchical scope, curated over time. It is standalone, yet borrows agntz's own brain — its tagger and curator are agntz YAML manifests, run identically by the TypeScript and Python SDKs.

Problem & tenets

The closest existing primitives — SessionStore (verbatim history) and ContextStore (named scratch buckets) — are short-term and uncurated. memrez adds durable, deduped, queryable knowledge that accumulates across sessions.

Small entries + tags

Memory is many small facts, each tagged with topics — not one Markdown blob per topic. Tags are the index.

The agent navigates

A cheap scan returns a table of contents; the agent reads only the topics it actually needs, instead of loading everything.

Cheap writes, heavy curation

A fast model tags on write; a stronger model dedupes and reconciles contradictions in the background. The thinking happens in curate, off the hot path.

Tags-first

Deterministic scan→read over topics now. Embeddings and semantic search come later, as an additive strategy.

One brain, two runtimes

The tagger and curator are agntz YAML manifests — identical behavior in the TypeScript and Python SDKs, the same way agntz already keeps parity.

Context & resources

Memory is not a one-off — it is the first resource an agent reaches through a shared, namespaced context. The foundation is intentionally generic so RAG, files, and future resources can use the same slot; memrez simply defines the first resource-specific interpretation of those grants. This agntz-core foundation must be built first (Phase 1A-1E).

resources: — what (design-time)

A manifest block, sibling to skills / reply, listing the resource kinds an agent may touch and how: per-agent mode (read | read-write), optional static namespace input, and per-kind options like autoScan or memory's writePolicy. Memory is just kind: memory — not a special block.

context: — where (run-time)

An array of granted namespaces passed to invoke(id, input, { context }) — the run's generic namespace capability set. A first-class invoke option, not a key in the free-form toolContext bag, so the runner threads it through every sub-agent narrow-only.

Grant definition. A grant is a normalized namespace string minted by trusted application code and passed at runtime through context. It is not a model argument and not a stored memory; it is a capability boundary that tells each resource provider the maximum namespace space this run may operate inside. agntz carries and narrows grants generically; each provider decides what operations those grants permit. 

# agents/support.yaml — declare resources (memory is a kind)
resources:
  memory:                                  # kind inferred from the name; memrez-backed
    mode: read-write                       # memory default; read-only registers no write tool
    autoScan: true                         # inject the scan TOC before the model call
    writePolicy:
      descendants: true                    # allow writes at the grant and below
      ancestorPromotion: none              # none | parent | ancestors; exact ancestors only
  product-docs:
    kind: rag
    mode: read                             # read-only FOR THIS AGENT
    namespace: gymtext/kb/product-docs     # static provider input, not memory semantics
// route — mint the grant from the authenticated user (the trust boundary)
const granted = `gymtext/user/${authedUserId}`;
await client.agents.run("support", message, { context: [granted], sessionId });
// effective resource input = manifest static namespaces ∪ per-run context
// each provider interprets that input under its own policy

Static namespaces are not automatic grants. A manifest namespace is provider input, not a universal read/write capability. RAG may treat it as a read corpus, files may treat it as a path prefix, and memory usually should avoid static namespaces in normal user-facing agents. agntz core passes the value through; the provider decides whether it is readable, writable, both, or just configuration.

The model never names a namespace. Tools are read(topic) / write(fact), never read(topic, namespace). The namespace stays off the model's tongue, so a prompt-injection has nothing to target. Narrowing to a sub-namespace is an app / sub-agent decision, never a model tool-arg.

You write through a resource, not to a namespace. For memory, a grant such as a/b/c reads a, a/b, and a/b/c; writes default to a/b/c and a/b/c/*. Optional promotion can allow exact ancestor writes such as a/b, but never a/b/*. memrez chooses the actual write scope and then validates it deterministically against the grants before storage. mode lives on the resource per agent, so the same corpus is read-only to one agent, read-write to another.

Context comes before memory. The first-class context option, narrow-only threading through invoke / agent-as-tool / spawn_agent, the resources: block, and the provider slot are the agntz-core foundation memrez builds on. They land as Phase 1A-1E before memrez-core. This also subsumes the old ContextStore / contextIds path (no real consumers today), freeing the word "context" for the grant.

Concepts & data model

One memory is one entry, with two independent axes: a single owning scope (the silo) and many-to-many topics (the tags within it).

type Scope = string; // "global" | "org/acme" | "org/acme/user/u_123" | ".../session/s_abc"

interface MemoryEntry {
  id: string;                 // mem_<nanoid>
  scope: Scope;               // single owning scope
  content: string;            // one canonical fact ("Prefers metric units.")
  topics: string[];           // semantic tags within the scope
  type: "fact" | "preference" | "event" | "summary";
  source?: { agentId?: string; sessionId?: string; runId?: string };
  status: "active" | "superseded";
  supersededBy?: string;      // id of the entry that replaced this one
  createdAt: string;
  updatedAt: string;
  // embedding?: number[];    // Phase 6
}

interface TopicSummary {
  topic: string;
  count: number;               // live count of active entries
  blurb?: string;              // curate-maintained one-liner
  lastUpdatedAt: string;
  hasUncuratedWrites: boolean; // writes since the last curate
}

Scope resolution. read / scan over org/acme/user/u_123 see entries at that path and every ancestor prefix (org/acme, global), so org-wide and global facts surface for a user without being copied. Each entry owns exactly one scope; moving a fact up a level (session→user) is a curate operation, not multi-membership.

Public API

The four operations are plain methods (also what the tools and the cron job call):

import { createMemrez } from "memrez";

const memrez = createMemrez({
  store: sqliteStore("./memory.db"),       // default: in-memory
  reasoner: agntzReasoner({ client }),      // default impl wraps the agntz SDK + the two manifests
});

const grants = ["org/acme/user/u_123"];

const toc     = await memrez.scan(grants);
const entries = await memrez.read(grants, "billing", { limit: 20 });
const result  = await memrez.write(grants, "Prefers email over phone.", {
  source: { sessionId, agentId },
});
const report  = await memrez.curate(grants);

Hosting is a one-line swap. The same interface — scan / read / write / curate / provider() / tools() — backed by a remote service instead of an in-process store + reasoner: 

import { createMemrezClient } from "memrez/client";

const memrez = createMemrezClient({ url: process.env.MEMREZ_URL, apiKey: process.env.MEMREZ_KEY });
// identical surface, now over HTTP — mem.tools() still returns kind: local handlers
type NamespaceGrant = string;

type WritePolicy = {
  descendants?: boolean;                       // default true
  ancestorPromotion?: "none" | "parent" | "ancestors";
};

scan(grants: NamespaceGrant[], opts?: { includeAncestors?: boolean; topicLimit?: number })
  : Promise<{ grants: NamespaceGrant[]; topics: TopicSummary[] }>;

read(grants: NamespaceGrant[], topic, opts?: { limit?: number; includeAncestors?: boolean })
  : Promise<MemoryEntry[]>;

write(grants: NamespaceGrant[], content, opts?: {
  type?: EntryType;
  topicsHint?: string[];
  source?: Source;
  writePolicy?: WritePolicy;
})
  : Promise<{ entry: MemoryEntry; action: "appended" | "superseded" | "deduped" }>;

curate(grants: NamespaceGrant[], opts?: { topics?: string[]; includeDescendants?: boolean })
  : Promise<CurateReport>;

Write-target validation is deterministic. If the memrez write agent proposes a namespace outside the configured write policy — for example a/b or a/b/d from grant a/b/c with default policy — memrez rejects it before storage. The provider can retry the internal write agent with the validation error or return a structured write-scope error.

Using it from an agntz agent

Memory is reached through the generalized resources: block — a sibling to agntz's existing skills and reply blocks. Memory is one resource kind; declaring it makes the runner register that kind's system tools (memory_read / memory_write) and, when autoScan is on, inject the scan TOC before the model call. 

# agents/support.yaml
resources:
  memory:                # kind inferred from the name; read-write + autoScan by default
    mode: read-write     # read-only | read-write; read-only registers only memory_read
    autoScan: true       # inject the scan TOC before the model call (the non-tool half)
    writePolicy:
      descendants: true
      ancestorPromotion: none
  # no namespace here — the grant is per-run, from the context option (never baked in)

You wire the implementation once, at app start, into the resources slot for memory. mem is a memory provider — embedded or a hosted client — and the YAML above never changes: 

const mem = createMemrez(options);          // or createMemrezClient({ url })

const client = await agntz({
  agents: "./agents",
  resources: { memory: mem.provider() },    // fills the memory slot
  tools: [...appTools],                     // memory_read / memory_write are NOT hand-listed
});
// load-time: an agent with a `memory` resource but no provider here → fail-fast, naming the agent

Locally, that's the whole setup: the batteries-included entry point (@agntz/runner) auto-wires memrez for kind: memory — an in-memory store + a fake or real reasoner, no API key needed offline. agntz core never bundles memrez; it only defines the slot the provider fills.

The grant comes from context, never the model. An agent cannot reach another user's memory by naming a scope — the tools are read(topic) / write(fact), with no namespace argument. The runner passes the run's context grant set to the memory provider, and autoScan scans that same grant set. The model only picks the topic or supplies the fact. 

type ResourceToolContext = {
  resourceName: string;
  mode: "read" | "read-write";
  config: unknown;                 // provider-specific options, e.g. memory.writePolicy
  grants: string[];                // normalized namespace capabilities
  run: { runId?: string; sessionId?: string; agentId?: string };
};

The grant threads to sub-agents automatically. Because context is a first-class invoke option (not a free-form toolContext value), the runner carries it through invoke, agent-as-tool, and spawn_agent narrow-only — a child sees its parent's grant or a narrowed subset, never a wider one. Resource mode clamps too: a child cannot regain write access if the parent invocation is effectively read-only. The same option is added to @agntz/sdk's agents.run(), retiring the old toolContext-passthrough gap.

Namespace grants as capabilities

A scope is where a memory is filed; a namespace grant is the capability boundary that decides which scopes a resource may touch. The isolation model is one rule:

The namespace grant is minted at the trust boundary from verified identity and is immutable for the request. The model never names or widens it; any narrowing to a sub-namespace is an app / spawning-parent decision, applied narrow-only — never a jump sideways.

So the grant set comes from auth (the API route, or a hosted workspace API key) and the model never controls or names it. Any narrowing to a sub-namespace is decided by the app or by a spawning parent — applied narrow-only, server-side — not emitted by the model. 

effective_grants = normalized(manifest static namespaces ∪ per-run context)
provider rules decide what read/write operations those grants permit

This is the discipline agntz already uses for ctx.user.id: the route loads identity and injects it via toolContext; the LLM chooses what to do, not whose data. A scope path is <tenancy…>/<subject> — leading segments are the namespace (app / workspace / org), the trailing segment is the subject (user / account). The exact shape is app-defined via a scope resolver, since only the app knows its tenancy model.

Memory reads up; writes stay down by default. For grant a/b/c, memrez may read a, a/b, and a/b/c, but never sibling branches. Writes default to a/b/c and a/b/c/*. If promotion is enabled, it can target exact ancestors such as a/b, but never a/b/*.

Example — gymtext (app → user)

gymtext                       ← trainer-wide memory (add later — costs nothing now)
└─ user/
   ├─ u_123   ← travel prefs, equipment access, goals, injuries
   └─ u_456   ← isolated: u_123's agent can never read this (sibling, not an ancestor)
// grant minted HERE, from the authenticated user — the trust boundary
const scope = `gymtext/user/${userId}`;

const result = await runner.invoke("chat", message, {
  sessionId,
  context: [scope],            // the namespace grant; threads narrow-only to sub-agents
  toolContext: { user },       // identity only; resources receive the context grants
});
// autoScan injects the scan TOC for `scope` before the model call — no manual prepend

memory_read / memory_write receive the run's context grant set; the model picks a topic but never a scope, so u_123 and u_456 can never see each other. Add trainer-wide memory later by trusted promotion to the exact gymtext scope — every user's read inherits it, with no per-user duplication and no schema change. Do not grant gymtext to a normal user-facing run, because default write semantics would include its descendants.

Example — sales platform (app → org → account)

sales
└─ org/
   ├─ acme                    ← org heuristics: "if property A is true, also find B"
   │  └─ account/
   │     ├─ a_789  ← "usually omits property A; assume X, then confirm"
   │     └─ a_790
   └─ globex                  ← different org: fully isolated from acme

The lead agent for a_789 is granted sales/org/acme/account/a_789; its reads inherit sales/org/acme (the org's qualification rules) but globex is invisible. Promotion is the curator's job: a trusted background curator deliberately granted sales/org/acme can inspect that org subtree, notice the same pattern across sibling accounts — "most omit property A" — and write a generalized heuristic at the exact org scope, where every account agent inherits it. Account quirks stay low; learned generalizations rise.

Running memrez: local vs hosted

"local" means two different things — separating them dissolves most of the confusion. Tool kind (kind: local) is how an agntz tool's handler is wired: an in-process function, versus kind: http (remote endpoint) or kind: mcp (MCP server). Topology is where memrez's store and reasoner physically live: embedded in your process, or a hosted service. They are independent — and the design pins the first and varies only the second.

The agent never changes; the provider does. You inject one memory provider via the memory resource slot. Whether mem is an in-process createMemrez(...) or a createMemrezClient({ url }) talking to a hosted service, the resources: block, the system tools, and your route code are identical — the same "graduate by changing one line" move agntz makes from @agntz/sdk to @agntz/client.

Why the system tools are local handlers. The runner-registered memory_read / memory_write run in-process before anything hits the wire: they pass the normalized context grant set to the injected provider. A raw kind: http tool pointed straight at hosted memrez would instead hand request-shaping to the model and force scope through templated headers — more surface, weaker isolation. So the tools stay local; when the provider is a hosted client, that call is just a thin HTTP hop.

The grant model is identical in both — what changes is who is trusted to mint the namespace grants and where they are enforced. Path shape, inheritance, and isolation never change. gymtext (one app, its own DB) is naturally embedded; a multi-org sales SaaS is the case for hosted, because one process then holds many tenants' memories.

There is no separate "memrez agent server." The reasoner (tagger / curator) lives inside whatever mem you construct. Embedded, it runs in your process. Hosted, it runs inside the memrez service, which embeds the agntz SDK to execute the two manifests and owns the curate schedule. One service owns storage + reasoner + cron; you call it over HTTP.

The background curator is grant-bound too. A hosted deployment runs the curator as a scheduled per-scope job; the curator for org/acme is deliberately granted the acme subtree and can never read another tenant. Every memrez operation — foreground tool call or background curate — runs inside a validated grant set. A curator that could read across grants would be a cross-tenant leak.

Reasoning layer (the shared brain)

Two agntz manifests, loaded by both runtimes from one canonical location (packages/memrez/agents/):

memrez-tagger · cheap model

In: { content, grants, existingTopics, writePolicy, topicsHint? }. Out: { namespace, topics, type, normalizedContent, duplicateOf? }. Drives write; memrez validates the proposed namespace before storage.

memrez-curator · stronger model

In: a validated grant set, a scope slice, and the topic set. Out: { ops: CurateOp[] } — merge, supersede, retag, setBlurb, promoteScope. Drives curate.

memrez-core only knows the Tagger / Curator interfaces. agntzReasoner() is the default implementation that runs these manifests through client.agents.run(...); the per-language ModelProvider (AI SDK / LiteLLM) makes the actual LLM call. Swap in a custom reasoner and memrez has no agntz dependency at all.

Storage

MemoryStore is memrez's own small interface (mirrors how agntz ships MemoryStore / SQLiteStore):

putEntry · getEntry · supersede(ids, byId)
listTopics(scopePaths) → TopicSummary[]            // aggregates topic + count + blurb
getByTopic(scopePaths, topic, limit) → entries
getTopicMeta / setTopicMeta(scope, topic, blurb, lastCuratedAt)
listScopeSlice(scope, opts) → entries              // for curate

Tables: entries, topic_meta (scope, topic, blurb, lastCuratedAt). TypeScript v1 adapters are in-memory + sqlite. Python currently has the in-memory memrez store only; sqlite is the first Python parity gap. Postgres + a vector column arrive with the embeddings phase and should be implemented for both runtimes when that phase starts.

scan TOC tuning (recommendation). Return all topic names + live counts (cheap even at hundreds of topics) plus curate-maintained blurbs; if topics exceed a cap, send blurbs only for the most-recently-touched K (names + counts for the rest). hasUncuratedWrites flags drift since the last curate. v1 is query-agnostic (the full map); query-aware ranking waits for embeddings.

Parity & contracts

Add contracts/memrez/ mirroring contracts/python-port/: scenario fixtures (a sequence of writes → expected topic assignment, write-target validation, supersede graph, scan TOC shape, scope-ancestor resolution). Both runtimes test against them.

Contract tests use a deterministic fake reasoner (a fixed content→topics mapping). That pins what must match across languages — storage, grant validation, scope resolution, dedup, supersede, scan aggregation — without asserting on nondeterministic LLM output. Real tagger/curator quality is covered separately by agntz evals on the two manifests.

Implementation phases

Phase 1A

namespace grant primitive · agntz-core change · prerequisite

Add the generic context: [namespaces] invoke option and normalize it into a run-level grant set. Define the shared namespace rules once: no leading/trailing slash, no empty segments, no traversal, no wildcards, deterministic ancestor/descendant helpers, and stable error messages. This phase does not add resources yet; it only creates the capability primitive memrez and future resources will consume. Start with table-driven tests for valid paths, invalid paths, ancestor checks, descendant checks, and sibling rejection.

Phase 1B

grant propagation & clamping · runner + SDK

Thread normalized grants through invoke, agent-as-tool, spawn_agent, and @agntz/sdk's agents.run(). Sub-agents may receive the parent's grant set or a narrowed subset only; attempts to widen or jump sideways fail before the child runs. Add focused tests for parent→child propagation and rejection of invalid narrowing.

Phase 1C

resources: manifest schema · manifest change

Add the generalized resources: block in @agntz/manifest: resource kind, per-agent mode (read | read-write), optional static namespace input, and provider-specific config passthrough. Parse-time validation checks shape, mode values, duplicate resource names, and reserved system tool names. Provider tool names are generated deterministically from the resource name plus provider tool name (for example memory_read or product_docs_search) so app tools cannot collide with system tools. No memory behavior belongs in this layer.

Phase 1D

generic resource provider runtime · core wiring

Add the generic provider slot in @agntz/core. At load time, fail fast if an agent declares a resource kind with no provider wired, naming the agent and kind. At run time, pass providers a ResourceToolContext containing resource name, effective mode, config, normalized grants, and run metadata. Effective mode clamps across parent→child calls so a child cannot regain write access. Keep the provider contract minimal and prove the slot with a fake provider, not memrez, before adding memory-specific behavior.

Phase 1E

cleanup, contracts, and docs · harden before memrez

Audit references to the unused ContextStore / contextIds path, then deprecate or remove it based on actual usage. Add contract fixtures for namespace normalization, grant narrowing, resource-mode clamping, provider load failures, and deterministic system tool naming, and document the generic provider boundary. The Python runtime mirrors the same contracts in Phase 6, but the fixture shape is established here.

Types, MemoryStore + in-memory adapter, grant validation, scope/ancestor resolution, scan/read/write/curate orchestration against injectable Tagger / Curator. Tests with a fake reasoner. Cheapest way to find out if the scope/topic model feels right in real code.

Fill the Phase 1D provider slot: register memory's system tools and honor autoScan through the resources framework; mem.provider() fills the slot, and the batteries-included entry point auto-wires it as the default memory provider. One end-to-end example agent.

Persistence across restarts; topic_meta table.

Phase 6

Python parity catch-up

Status: partial. Python already mirrors namespace grants, context passthrough, resource manifest parsing, and the in-memory memrez core. Finish parity by adding the missing runtime and storage pieces below, then wire contracts/memrez/ into both test suites.

  • Add the Python generic resource provider runtime: provider lookup, mode clamping, resource context injection, provider tool registration, and provider-missing failures.
  • Add the Python memory resource provider: memory_read, memory_write, auto-scan context injection, read-only omission of write tools, and source metadata.
  • Add SqliteMemoryStore for Python memrez with the same entry, topic index, supersede, source JSON, and topic metadata behavior as TypeScript.
  • Add a Python agntz-backed memrez reasoner that runs the same memrez-tagger.yaml and memrez-curator.yaml manifests, or packaged copies of them.

Open questions

All have a recommended lean; none block the early phases.

  1. Auto-injection. Read-side is resolved — the memory.autoScan flag makes TOC injection a declarative opt-in (the runner scans and prepends). Open sliver: should the runner ever auto-write (post-run fact extraction)? Lean: no — the caller passes finished facts (out of scope for v1).
  2. write dedup depth. How hard does the cheap tagger try to detect "already known" / supersede vs. always append and let curate clean up? Lean: append + exact-dup skip only; curate does the rest.
  3. Promotion trigger. session→user, or cross-sibling→parent (the sales "most accounts omit A" case). Curator-driven is the primary path; do we also expose an explicit promote() API? Promotion validates exact ancestor targets only, never ancestor subtrees. Lean: curator-driven, explicit API as an escape hatch.
  4. Memory write policy shape. Default is descendants: true and ancestorPromotion: none. Trusted resource agents may opt into parent or ancestors promotion, but normal foreground agents should not. Lean: ship the policy object now, keep non-default promotion behind explicit config.
  5. Blurb storage. topic_meta table vs a type:"summary" entry per topic. Lean: topic_meta.
  6. Package name. memrez is a separate package you inject as the provider — bare memrez (honors standalone) vs @agntz/memrez (monorepo consistency; signals the tight slot coupling). Doc assumes bare memrez.
  7. Where the slot lives. The resources: schema goes in @agntz/manifest; does the generic ResourceProvider interface live in @agntz/core (runner-adjacent, like SkillStore) or a thin shared contract package? Lean: alongside SkillStore in core; memory's provider contract can live in memrez.
  8. Manifest location across languages. Canonical YAML in packages/memrez/agents/; does Python read those files directly or get a packaged copy at build?
  9. SDK context passthrough (agntz-side) — resolved. Superseded by Phase 1A and 1B: the per-run grant is a first-class context: [namespaces] invoke option that the core Runner threads narrow-only, and the same option is added to @agntz/sdk's agents.run() — built before the memory resource, so no memory-specific toolContext passthrough is needed.

Out of scope (v1)

  • Embeddings / semantic search (Phase 7).
  • Extracting facts from raw transcripts (the caller's job for now).
  • Multi-scope-per-entry membership.
  • A hosted memrez service or UI.
  • Cross-tenant / global curate runs (curate operates inside one validated grant set).
