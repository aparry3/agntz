Python SDK and libraries for Agntz

Agntz is TypeScript-first today. The Python port should let a Python developer read the same agent YAML, call agents locally or through the hosted client, and get started as quickly as the TypeScript docs allow.

What exists today

Agntz has two surfaces that matter for Python: the hosted HTTP client and the embedded local SDK. They should share public concepts but remain separate internally.

Hosted client

@agntz/client is a small HTTP client around the worker API. It exposes agents, runs, traces, health checks, auth headers, typed errors, cancellation, and SSE streaming.

Low risk Immediate value HTTP parity

Embedded local SDK

@agntz/sdk loads a directory of YAML manifests, registers local tools, creates an in-process runner, and exposes the same high-level client.agents.run and client.agents.stream shape.

Developer experience

Target Python API

The Python API should feel recognizable to someone reading TypeScript docs, but use Python naming, Python data models, and Python execution conventions.

Local quickstart

from agntz import agntz

client = agntz(agents="./agents")

result = client.agents.run(
    agent_id="support",
    input="Help me debug this invoice",
)

print(result.output)

Hosted quickstart

import os
from agntz import AgntzClient

client = AgntzClient(
    api_key=os.environ["AGNTZ_API_KEY"],
    base_url="https://api.agntz.co",
)

result = client.agents.run(
    agent_id="support",
    input="Hello",
)

Python local tools

from pydantic import BaseModel
from agntz import agntz, tool

class AddInput(BaseModel):
    a: float
    b: float

def add(args: AddInput) -> dict:
    return {"result": args.a + args.b}

math_tool = tool(
    name="add",
    description="Add two numbers",
    input_schema=AddInput,
    execute=add,
)

client = agntz(agents="./agents", tools=[math_tool])

Streaming

for event in client.agents.stream(
    agent_id="support",
    input="Walk me through the fix",
):
    if event.type == "reply":
        print(event.reply.content)
    elif event.type == "complete":
        print(event.result.output)

Compatibility rule

Public method names should be Pythonic, such as agent_id and session_id. Wire output should remain serializable to the hosted contract, such as agentId and sessionId. This can be handled with Pydantic aliases or explicit model serializers.

Package design

Recommended architecture

Ship one PyPI package named agntz. Keep the internal modules close enough to the TypeScript package boundaries that future parity work is easy to reason about. 

agntz/
  __init__.py           # agntz(), tool(), AgntzClient, AsyncAgntzClient
  client/               # hosted HTTP client, resources, SSE, errors
  manifest/             # YAML models, parser, validation, execution helpers
  core/                 # Runner, tool registry, model provider protocol
  stores/               # MemoryStore first, SQLite later
  sdk/                  # embedded local client construction
  cli/                  # agntz command for validation and local runs

Client layer

Use httpx for HTTP and implement a small SSE parser or a thin wrapper that preserves the current event names and error behavior. Provide sync and async clients.

Manifest layer

Use Pydantic models and a canonical JSON Schema. Preserve state initialization, template interpolation, condition semantics, input transforms, and output mappings.

Runtime layer

Use Python protocols for ModelProvider, Store, and tool execution. Start with memory storage and add SQLite once the local runner stabilizes.

Execution plan

Migration phases

0

Freeze the contracts

Create a canonical manifest schema and shared fixture suite before writing much Python code. This prevents two runtimes from drifting while both are moving.

  • Add shared YAML fixtures for every agent kind.
  • Add expected normalized manifests, state snapshots, and output mappings.
  • Pin stream event shapes and hosted client error behavior.
  • Fix known TS local parity gaps before Python copies them.
1

Ship the hosted Python client

This gives Python users immediate value without duplicating the runner. The Python client should call the same worker endpoints as the TS client.

  • Implement AgntzClient and AsyncAgntzClient.
  • Implement agents, runs, traces, health, errors, timeouts, and abort/cancellation behavior.
  • Implement SSE parsing and stream cleanup semantics.
  • Add tests mirroring packages/client/tests.
2

Port the manifest package

This is the heart of identical YAML behavior. It should be a close semantic port, not a new interpretation of the schema.

  • Use Pydantic for manifest models and validation.
  • Port template interpolation, conditions, state initialization, transforms, and output mapping.
  • Preserve JavaScript-like edge cases that are part of today's YAML behavior, or explicitly change both runtimes together.
  • Run TS and Python against the same fixtures in CI.
3

Build the embedded runner MVP

Add local agent execution, local Python tools, MemoryStore, session handling, and model provider protocols.

  • Expose agntz(agents="./agents", tools=[...]).
  • Support LLM, tool, sequential, and parallel manifests.
  • Support local run and trace buffers equivalent to the TS embedded SDK.
  • Add OpenAI, Anthropic, OpenRouter, and custom model providers behind one protocol.
4

Add advanced parity

Once the core local path is stable, port features that need more runtime integration.

  • HTTP tools with auth, token refresh, redaction, and request templating.
  • MCP tools and transport support.
  • Spawnable agents, reply events, multimodal content, and structured output handling.
  • SQLite store, tracing hooks, eval hooks, skills, and secrets as the TS embedded surface supports them.
5

Document and release

Python users should be able to install, load YAML, run an agent, stream a result, add a local tool, and switch to hosted in minutes.

  • Publish agntz to PyPI with extras such as agntz[sqlite], agntz[mcp], and provider extras if needed.
  • Add Python tabs to the quickstart, tools, streaming, sessions, and hosted-client docs.
  • Include examples that share the same YAML between TS and Python.
  • Add a release checklist that runs TS and Python contract tests together.
Quality bar

Parity testing strategy

The migration succeeds only if the same YAML has the same observable behavior in both runtimes. Unit tests are not enough; this needs contract fixtures shared across languages.

Shared golden fixtures

  • Manifest parse and normalized model output.
  • Validation success and failure messages for references, tools, schemas, and provider config.
  • Template rendering and nested {{#if}} blocks.
  • Condition expressions, including equality, inequality, &&, and ||.
  • State initialization, input transforms, output mappings, and state keys.

Runtime contract tests

  • Mock model provider for deterministic LLM outputs and tool calls.
  • Local tool dispatch with successful calls and fail-fast missing tools.
  • Sequential and parallel pipeline execution snapshots.
  • HTTP tool URL, headers, body, auth, truncation, and redaction behavior.
  • Run and trace list/get/stream behavior for hosted and local clients.

CI recommendation

Add a top-level contract test command that runs TypeScript fixtures and Python fixtures against the same files. When a manifest feature changes, both languages should fail until the contract is intentionally updated.

Docs plan

Python docs should mirror the TS path

The current TypeScript quickstart sells the right idea: define YAML, call agntz(), then swap to the hosted client later. Python docs should use the same structure with Python tabs and examples.

Quickstart

pip install agntz, create one YAML agent, run locally, print output.

Local tools

Use Pydantic models and tool(...) so Python developers do not need to hand-write JSON Schema.

Hosted swap

Replace agntz(...) with AgntzClient(...); keep client.agents.run usage recognizable.

Docs cleanup first

Some TS docs appear ahead of the live SDK surface. Reconcile the current TS docs with the actual implementation before adding Python docs, so the Python package does not inherit stale promises.

Risks and decisions

Important calls before implementation

Do not port the whole app first

Hosted Python should call the existing worker. Rebuilding the app or hosted worker in Python is not required for the SDK goal and would delay useful client parity.

Do not depend on Node for local Python

A subprocess bridge to the TS runner could be a temporary validation tool, but the released embedded SDK should be Python-native.

Provider parity is the hardest runtime area

The TS runner gets tool calling and structured output through the Vercel AI SDK. Python should isolate provider differences behind a ModelProvider protocol and use fixture-driven tests.

Template and condition semantics must be pinned

If Python uses normal Python truthiness and equality without tests, YAML can behave differently. Decide whether to emulate current TS behavior or change both runtimes together.

Known gap Why it matters Recommendation
Embedded TS local SDK appears to drop manifest outputSchema during manifest-to-agent conversion. Hosted worker and embedded local execution can diverge for structured output. Fix or explicitly test this before Python copies the behavior.
Top-level HTTP tool validation appears stricter than LLM HTTP tool validation. Same YAML feature can be allowed in one position and rejected in another. Normalize schema rules before freezing the manifest contract.
Docs reference some methods/options that are not present in the live local SDK. Python docs should not replicate outdated TS promises. Audit docs and implementation together in Phase 0.

This planning page is intentionally implementation-neutral. It defines the migration shape, parity bar, and sequencing for a Python SDK that can share agent YAML with the TypeScript runtime.
