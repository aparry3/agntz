TypeScript Formatter & Lint
Tooling plan

One formatter, one linter, one gate: finish adopting biome.

biome is already a dependency in six of eight TypeScript packages, but it has no config and nothing enforces it. This plan commits a root config matched to the existing style, brings every package and the two Next.js apps under it, and wires it into CI and pre-commit so it can never drift again — landed as a single PR.

Current State

The “269 lint violations” are mostly an illusion: biome runs on its default config (tab indentation) while the codebase is 2-space. Point a config at the real style and the bulk disappears, leaving import-sorting plus a handful of genuine rule fixes.

Fact State today
biome dependency Present in core, sdk, manifest, client, worker, memrez (6/8). Missing in store-sqlite, store-postgres.
lint script Wired (biome check src/) in 5 packages; memrez has the dep but no script; stores have neither.
biome config None anywhere. Runs on pure upstream defaults the code was never written to.
Other formatters No Prettier, no .editorconfig. biome is the only tool installed.
Existing style 2-space indent, double quotes, semicolons. (biome default: tab, double, semicolons, 80-col.)

CI enforcement None. The job named “Lint & Type Check” runs build + turbo typecheck only — it never runs pnpm lint.
Violation profile ~269 across 5 packages: dominated by indentation (tab vs 2-space) + import-sorting. Real rule violations in source: 4 (see Real Fixes).

Decision

Standardize on biome as the single formatter and linter for all TypeScript. The choice is effectively already made — it is installed and partway adopted. Do not introduce a Prettier + ESLint pair alongside it.

One tool

biome is formatter and linter in a single fast binary. Replaces the Prettier + ESLint + eslint-config-prettier dance with one root config.

Already chosen

Six packages already depend on it. The work is not picking a tool — it is giving the one we have a config and a gate.

Low churn

Matched to existing style, the reflow is small: indentation already matches once configured; quotes and semicolons already conform.

Root Config

A single biome.json at the repo root. biome walks up from each package to find it, so per-package lint scripts keep working and resolve against this one file. 

// biome.json (repo root)
{
  "$schema": "https://biomejs.dev/schemas/1.9.4/schema.json",
  "organizeImports": { "enabled": true },
  "formatter": {
    "enabled": true,
    "indentStyle": "space",
    "indentWidth": 2,
    "lineWidth": 100
  },
  "javascript": {
    "formatter": {
      "quoteStyle": "double",
      "semicolons": "always"
    }
  },
  "linter": {
    "enabled": true,
    "rules": {
      "recommended": true,
      "style": {
        "noNonNullAssertion": "off"
      }
    }
  },
  "files": {
    "ignore": ["**/dist/**", "**/.next/**", "**/node_modules/**", "**/*.gen.ts"]
  },
  "overrides": [
    {
      "include": ["**/tests/**", "**/__tests__/**", "**/*.test.ts"],
      "linter": { "rules": { "suspicious": { "noExplicitAny": "off" } } }
    }
  ]
}
Setting Value Why
indentStyle / width space / 2 Matches existing code. This single setting erases the bulk of the 269.
quoteStyle double Already matches (biome default). Zero churn.
semicolons always Already matches. Zero churn.
lineWidth 100 Modern default; less wrap churn than biome’s 80. Tune after seeing the diff.
organizeImports on Sorts imports deterministically — the second-largest violation category.
noNonNullAssertion off Noisy, mostly fires in tests, and its autofix (!→?.) is behavior-changing — it silently no-ops instead of throwing, hiding test failures.
noExplicitAny (tests) off in tests any is acceptable in test scaffolding; keep it enforced in src.

Keep the rules with real signal enabled: noExplicitAny (in src), noAssignInExpressions, useLiteralKeys. The point of the gate is to catch those, not to relitigate brace style.

Next.js Apps

biome formats, ESLint guards

biome formats and lints app and site like everything else. A thin ESLint config keeps only Next’s framework rules (image/script/web-vitals) that biome does not replicate.

Minimal eslintrc

Each app gets { "extends": "next/core-web-vitals" } — which they need anyway to stop the interactive setup prompt. No formatting rules in ESLint; formatting is biome’s job. 

// packages/app/.eslintrc.json  (and packages/site/.eslintrc.json)
{ "extends": "next/core-web-vitals" }

Rejected alternative: dropping next lint entirely for an all-biome setup. Simpler, but it loses Next’s accessibility and performance rules. The one extra config file is worth keeping them.

Scope

All 8 TS packages

Add the biome devDependency and a lint script to store-sqlite, store-postgres, and memrez (currently unlinted). Consistency is the entire point.

Lint tests too

Point lint scripts at the package root (biome check .), not just src/, so test files are covered — with the test-file rule relaxations from the overrides block.

Two Next apps

biome for format + general lint; ESLint next/core-web-vitals for framework rules.

One root config

No per-package biome configs. Everything resolves against the single root biome.json.

Enforcement

A config does nothing on its own. Without CI + a hook, the repo drifts straight back to where it is today. Enforcement is the part that makes this stick.

CI gate

Add a real lint step to the “Lint & Type Check” job: pnpm turbo lint (or biome ci . at the root). It must actually fail the build on violations.

Pre-commit hook

lefthook (lighter than husky) running biome check --staged --no-errors-on-unmatched on staged files only — fast, local, catches drift before it lands.

Blame hygiene

Add .git-blame-ignore-revs with the reflow commit SHA and set git config blame.ignoreRevsFile. GitHub honors it automatically.

The Reflow

One commit

Run biome check --write (safe fixes only — not --unsafe). Land it as a single isolated commit: style: apply biome formatting across monorepo. No logic changes in this commit.

Lands with the gate

The reflow and CI enforcement must land together. Turn the gate on before reflowing and CI goes red; reflow without the gate and it drifts. One PR, sequenced commits.

Timing

Do it now: main is clean and in sync, and the window right after a merge — before new branches fan out — is the lowest-conflict moment for a repo-wide reflow.

In-flight branch caveat. A repo-wide format touches nearly every file and will conflict with anything outstanding. The provider-harness line is noted as carrying an unmerged core fix — confirm it and any other active branches are merged or paused first, or land the config + gate now and hold the reflow until the tree is quiet.

Real Fixes

Stripped of formatting noise, only four rule violations live in source — the only ones that might mask a real bug rather than a style preference. Read and fix these by hand, not by autofix, before the reflow.

Location Rule Action
core/src/pipeline/sequential.ts:71 noNonNullAssertion Confirm the value is truly guaranteed non-null; replace ! with a guard or keep with an intentional comment.
core/src/webhooks/dispatcher.ts:140 noNonNullAssertion Same — a ! in webhook dispatch could hide a missing-field case.
manifest/src/http-url.ts:38 noAssignInExpressions Likely an intentional exec() loop; confirm and make the assignment explicit if so.
manifest/src/http-url.ts:110 noAssignInExpressions Same pattern; verify it is not an accidental = for ===.

The other ~11 non-null assertions, the lone any, and the computed-key nits all live in test files and are handled by the config (rule off / test override) — no manual work.

Risks & Mitigations

Risk Mitigation
Reflow conflicts with in-flight branches Time it between branches on a quiet tree; or land config + gate first and hold the reflow. Rebase outstanding branches right after it merges.
Unsafe autofixes change behavior Use safe --write only. Hand-review the four real violations. Never run --unsafe in the reflow.
Blame noise from mass reformat .git-blame-ignore-revs pointed at the single reflow commit; honored by GitHub and local git.
Pre-commit friction slows commits lefthook runs biome on staged files only — near-instant. Auto-fixes on commit where safe.
lineWidth 100 still over-wraps It is one config line. Review the reflow diff; adjust before merging if a different width is calmer.
