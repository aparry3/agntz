TypeScript Formatter & Lint
Tooling plan

One formatter, one linter, one gate: finish biome coverage for TypeScript modules.

biome already runs cleanly in the five modules that have lint scripts. The gap is coverage: several TypeScript workspaces are skipped, the repo has no root policy, and CI never runs lint. This plan pins the current style, brings the TS modules and provider harness under the same gate, and leaves the larger Next.js UI lint debt for a separate pass.

Current State

pnpm lint is green today, but it only checks five workspaces. The current TypeScript style already matches biome defaults: tabs, double quotes, semicolons, and 80-column wrapping. Standardization is therefore a coverage and enforcement problem, not a repo-wide reformat problem.

Fact State today
biome dependency Present in core, sdk, manifest, client, worker, memrez. Missing in store-sqlite, store-postgres, app, site, and provider-harness.
lint script Root pnpm lint already runs turbo lint, but only client, core, manifest, sdk, and worker define a lint script. The other workspaces are skipped.
biome config None anywhere. The checked modules happen to match upstream defaults today; a root config should pin that intentionally so an upstream default cannot move the repo.
Other formatters No Prettier, no .editorconfig. biome is the only installed formatter/linter in TS packages.
Existing style Tabs in TypeScript, double quotes, semicolons, 80-column biome wrapping. JSON files use 2-space indentation.
Next.js apps app and site have no lint scripts. biome formatting mostly fits, but enabling the biome linter on them exposes ~173 existing UI diagnostics across hooks, a11y, non-null assertions, and array keys.
CI enforcement None. The job named “Lint & Type Check” runs build + turbo typecheck --filter="./packages/*" only. It never runs pnpm lint, and it excludes apps/provider-harness.
Violation profile The existing checked module source is clean. The unlinted stores and memrez source are clean. provider-harness has one real source violation: noImplicitAnyLet. Tests need targeted overrides if included.

Decision

Standardize on biome as the single formatter and linter for TypeScript modules. The choice is effectively already made — it is installed and passing where wired. Do not introduce a Prettier + ESLint pair alongside it.

One tool

biome is formatter and linter in a single fast binary. One root config is enough for the package modules and the provider harness.

Already chosen

Six workspaces already depend on it and five already pass. The work is not picking a tool — it is finishing coverage and making CI run it.

Low churn

Pinning biome defaults keeps churn low. The checked modules need no formatting reflow; only newly covered files and staged changes should be touched.

Root Config

A single biome.json at the repo root. biome walks up from each workspace to find it, so per-workspace lint scripts keep working and resolve against this one file. The config deliberately pins the style that already passes. 

// biome.json (repo root)
{
  "$schema": "https://biomejs.dev/schemas/1.9.4/schema.json",
  "organizeImports": { "enabled": true },
  "formatter": {
    "enabled": true,
    "indentStyle": "tab",
    "lineWidth": 80
  },
  "javascript": {
    "formatter": {
      "quoteStyle": "double",
      "semicolons": "always"
    }
  },
  "json": {
    "formatter": {
      "indentStyle": "space",
      "indentWidth": 2
    }
  },
  "linter": {
    "enabled": true,
    "rules": {
      "recommended": true
    }
  },
  "files": {
    "ignore": ["**/dist/**", "**/.next/**", "**/node_modules/**", "**/*.gen.ts"]
  },
  "overrides": [
    {
      "include": ["**/tests/**", "**/__tests__/**", "**/*.test.ts"],
      "linter": {
        "rules": {
          "style": { "noNonNullAssertion": "off" },
          "suspicious": {
            "noExplicitAny": "off",
            "noAssignInExpressions": "off"
          }
        }
      }
    },
    {
      "include": ["packages/app/**", "packages/site/**"],
      "linter": { "enabled": false }
    }
  ]
}
Setting Value Why
indentStyle / width tab / biome default Matches existing TypeScript. Switching to spaces would create a noisy, unnecessary reflow.
quoteStyle double Already matches (biome default). Zero churn.
semicolons always Already matches. Zero churn.
lineWidth 80 Matches current formatting. Raising this to 100 or 120 collapses many intentionally wrapped declarations.
organizeImports on Already clean in the checked modules; keep it enabled so imports stay deterministic.
JSON formatter space / 2 TS uses tabs, but package/config JSON files already use 2-space indentation.
test overrides any, non-null assertions, assignment loops relaxed These are common in test scaffolding and currently account for most test-only diagnostics. Keep the source rules strict.
Next app override formatter yes, linter no Lets staged app files be formatted without forcing the ~173 existing UI lint fixes into the module standardization PR.

Keep the rules with real signal enabled in source: noImplicitAnyLet, noExplicitAny, noAssignInExpressions, and useLiteralKeys. The point of the gate is to catch real source drift while keeping historical test patterns from blocking adoption.

App Boundary

Do not hide app debt

app and site should use the same formatter, but their linter adoption is not the same task as module standardization. Turning on biome’s recommended linter for both apps currently reports ~173 diagnostics.

Stage app lint separately

Handle hooks, accessibility, array keys, SVG titles, and app non-null assertions in a follow-up UI lint PR. Until then, disable biome linter diagnostics for app paths while still allowing formatting and import organization.

Path Initial gate Follow-up
packages/app Format and organize imports only. Fix existing React hook, a11y, SVG-title, key, and non-null diagnostics, then enable lint.
packages/site Format and organize imports only. One current import-sort fix is expected. Fix landing/docs UI diagnostics, then enable lint.

If the team still wants framework-specific Next rules, add that as a separate ESLint decision. Do not mix it into the TS module standardization PR; that would turn a small tooling fix into a UI lint migration.

Scope

All TS modules

Keep the five passing lint scripts and add coverage for memrez, store-sqlite, store-postgres, and provider-harness. These are the module workspaces currently skipped by pnpm lint.

Include tests

Prefer linting src plus tests for modules that have tests. Use targeted test overrides for any, non-null assertions, and assignment-in-loop patterns instead of weakening source rules.

Next apps later

Do not enable app lint in the first PR. Allow formatting/import sorting, document the current diagnostics, and land app lint once the UI debt is fixed.

One root config

No per-package biome configs. Everything resolves against the single root biome.json.

Enforcement

A config does nothing on its own. Without CI + a hook, the repo drifts straight back to where it is today. Enforcement is the part that makes this stick.

CI gate

Add a real lint step to the “Lint & Type Check” job: pnpm lint. It must run before or alongside typecheck and fail the build on violations.

Pre-commit hook

lefthook filters staged TypeScript module files, then runs biome check --write --no-errors-on-unmatched {staged_files} — fast, local, and scoped to the files being committed.

Root commands

Keep pnpm lint as the canonical check. Add pnpm format as a targeted biome check --write pass over packages/*/src, packages/*/tests, and apps/provider-harness/src.

Format Pass

No mass reflow

The existing checked modules already pass biome formatting with defaults. Do not run a repo-wide style churn commit unless the final config produces an actual diff worth isolating.

Targeted writes

Run biome check --write only on newly covered paths and staged files. Use safe fixes only; avoid --unsafe for the adoption pass.

Blame stays clean

If the final diff is just config, package scripts, one provider-harness type fix, and possibly one app import sort, skip .git-blame-ignore-revs. Add it only if a broad format commit becomes unavoidable.

Raising lineWidth to 100 or 120 looks tempting, but it causes biome to collapse many existing multiline declarations. Keep 80 unless the team explicitly wants a separate formatting change.

Real Fix

The unlinted module source is almost clean. The only source issue found while checking the newly covered module paths is in provider-harness.

Location Rule Action
apps/provider-harness/src/index.ts:171 noImplicitAnyLet Give results an explicit type or initialize it from the returned runMatrix shape. This should be a normal source fix, not a suppression.

When module tests are included, expect historical test scaffolding diagnostics: explicit any, non-null assertions, and one assignment-in-loop SSE parser. The proposed test override handles those without reducing source strictness.

Risks & Mitigations

Risk Mitigation
Accidentally creating a repo-wide reflow Pin tabs and 80-col. Run writes only on targeted paths. Review the diff before committing.
Unsafe autofixes change behavior Use safe --write only. Hand-fix the provider-harness type issue. Never run --unsafe in the adoption pass.
Next app diagnostics block module adoption Disable linter diagnostics for packages/app and packages/site in the initial config; track app lint as a follow-up PR.
Pre-commit friction slows commits lefthook runs biome on staged files only and writes safe fixes. Keep the command scoped and fast.
Test overrides hide source problems Apply the relaxed rules only under tests, __tests__, and *.test.ts. Source keeps recommended biome rules.
