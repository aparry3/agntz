Namespace Grant Security
Security plan

Keep namespace grants generic, but make grant issuance policy controlled.

Namespace grants stay as reusable runtime capabilities. A new resource-level policy layer decides which namespaces are legal to grant, which namespaces are structural only, and when elevated access is required.

Problem Security Model Resource Kinds Policy Shape Runtime Flow Examples Contract Tests

Problem

Namespace hierarchy is powerful

A grant like gymtext/private/users can be meaningful for trusted jobs, but it is too broad for normal user-scoped agent runs because it can expose every user branch below it.

The algorithm should not change

The current namespace algorithm still works: grants are normalized, reads can see allowed ancestors, and writes stay within allowed target namespaces. The missing piece is an issuance policy that prevents dangerous grants from being handed out.

Security Model

Namespaces

Define hierarchy and addressability for resource data. They are not security by themselves.

Grants

Represent runtime access capabilities issued by trusted app or runtime code.

Policy

Defines which grants are legal to issue for a resource, subject, and capability set.

Subject

Binds grants to authenticated state such as subject.userId, organization, tenant, or service identity.

Resource

Still validates each read and write operation against the issued grants before touching storage.

Resource Kinds

Memory is not a core keyword. A resource map key such as memory is the resource instance name and tool prefix. The resource kind is the provider capability type. When kind is omitted, it defaults to the resource name for shorthand.

Shorthand

resources:
  memory:
    mode: read-write

This normalizes to kind: memory because the resource name is memory.

Explicit kind

resources:
  privateMemory:
    kind: memory
    mode: read-write

  sharedMemory:
    kind: memory
    mode: read

This allows multiple memory resource instances with different names, modes, configs, and namespace policies.

Concept Example Purpose
Resource name privateMemory Agent-local instance name and model tool prefix, such as privateMemory_read.
Resource kind memory Provider capability type used to find a compatible resource provider.
Provider instance user-memory Optional future selector for a concrete store/provider when several providers share one kind.

Multiple memory resources are useful for separation and ergonomics, but they are not a substitute for namespace policy. Broad grants still need to be rejected even if an agent uses separate privateMemory and sharedMemory resource entries.

Policy Shape

This policy belongs to resource configuration, not to memrez specifically. The same shape can be reused later by memory, RAG, document stores, search indexes, or any other namespaced resource. 

resources:
  privateMemory:
    kind: memory
    mode: read-write
    namespacePolicy:
      structuralOnly:
        - gymtext
        - gymtext/private
        - gymtext/private/users

      grantable:
        - gymtext/public/**
        - gymtext/private/users/{subject.userId}

      elevatedGrantable:
        memory:admin:
          - gymtext/private/users/**
          - gymtext/private/users

  sharedMemory:
    kind: memory
    mode: read
    namespacePolicy:
      grantable:
        - gymtext/public/**

Structural only

These namespaces can exist as parents, but normal runs cannot receive them as direct grants. This prevents accidental tenant-parent access.

Subject bound

Placeholders like {subject.userId} are resolved from trusted authenticated state, not model output or agent-provided input.

Runtime Flow

Normalize

Normalize all runtime context grants before agent execution.

Reject

Reject grants listed in structuralOnly or outside normal grantable patterns.

Bind

Resolve subject placeholders from trusted request/session state and reject mismatches.

Elevate

Allow broad grants only when the caller has the required explicit capability.

Recheck

Resource tools continue to validate each read and write using the grants for that run.

Examples

Grant Subject Capabilities Result
gymtext/public/general any user none Allowed. Public resource branch.
gymtext/private/users/user1 user1 none Allowed. Subject-bound private branch.
gymtext/private/users/user2 user1 none Rejected. Subject mismatch.
gymtext/private/users user1 none Rejected. Structural namespace and tenant parent.
gymtext/private/users service identity memory:admin Allowed. Explicit elevated grant.

Normal user runs should receive narrow grants such as gymtext/public/general and gymtext/private/users/user1. Broad parent grants should be reserved for trusted service identities, admin flows, migrations, backfills, and controlled curation jobs.

Contract Tests

Positive cases

  • Allows gymtext/private/users/user1 for subject user1.
  • Allows gymtext/public/general for normal runs.
  • Allows broad parent grants only with the required elevated capability.
  • Allows future resources to use the same policy shape without memory-specific keys.

Negative cases

  • Rejects gymtext/private/users for normal user runs.
  • Rejects gymtext/private/users/user2 for subject user1.
  • Rejects grants outside configured grantable patterns.
  • Confirms resource read and write validation still runs after grant issuance.

Implementation Notes

Agents and models must never be trusted to mint broad grants. Grants come from trusted runtime code, and resource policies validate them before execution begins.

This should be additive to the current implementation: add policy validation ahead of execution, keep namespace narrowing and resource operation checks intact, and keep memrez-specific write decisions inside the memory resource.

The current provider lookup by kind is enough for multiple resource entries backed by the same memrez provider. If we need physically separate memory stores, add an optional provider or instance selector so kind: memory can resolve to named provider instances such as user-memory and org-memory.
