Context, resources, memrez
Plan for /docs and /llms.txt

Update the docs around namespace context, resource providers, and memrez before they drift.

The repo now has a generic resource system, runtime namespace grants, and a first memory provider. The rendered docs are still mostly agent/schema/SDK oriented, so both human readers and LLM ingestion are missing the current mental model.

Docs Source Map

Rendered docs and LLM corpus

The source of truth for /docs, sidebar navigation, per-page markdown, and /llms.txt is packages/site/src/components/docs/manifest.ts. Any new human page must be added there or the LLM corpus will miss it.

  • packages/site/src/app/docs/[[...slug]]/page.tsx renders pages.
  • packages/site/src/app/llms.txt/route.ts concatenates all docs pages.
  • packages/site/src/middleware.ts rewrites /docs/*.md to raw markdown.

Legacy markdown docs

The root docs/ tree contains useful material, especially docs/guide/07-context.md and docs/schema.md, but it is a separate VitePress surface. Treat it as a source to port or explicitly mark it legacy.

  • docs/guide/07-context.md has the current context vs contextIds distinction.
  • docs/schema.md already has a resource block reference to reuse.

Runtime implementation

The core runner validates namespace grants, resolves resources, injects provider context, and registers generated resource tools. Documentation should follow the runtime rather than older planning notes.

  • packages/core/src/namespace.ts defines grant normalization and protected namespace policy.
  • packages/core/src/types.ts defines ResourceProvider and ResourceToolContext.
  • packages/core/src/runner.ts clamps modes and registers resource tools.

Memrez implementation

Both TS and Python now expose memrez primitives and resource provider wiring. Some older planning text is stale; do not copy claims that Python lacks provider tools.

  • packages/memrez/src/provider.ts exposes read, write, and auto-scan.
  • python/src/agntz/memrez_provider.py mirrors the memory resource provider.
  • python/src/agntz/sdk/local.py resolves resource providers and executes generated tools.

Execution Phases

Phase 0

Lock the source-of-truth decision

Decide whether docs/ remains a maintained VitePress surface or becomes legacy/internal. The practical default should be: packages/site is public and LLM canonical, root docs/ is porting source until cleaned up.

  • Add a short maintainer note to docs/index.md or repo docs explaining canonical public docs.
  • Do not add new docs only under root docs/ unless they are also added to DOCS_GROUPS.
  • Capture a follow-up to remove or sync stale VitePress links such as old Studio pages.
Phase 1

Add the context and resources mental model

Create a concept page before adding memrez details. It should be clear that context is a trusted namespace capability array, not a prompt scratchpad, and that contextIds remains the legacy scratchpad API.

  • Port and tighten docs/guide/07-context.md into packages/site/src/components/docs/pages/concepts/context-and-resources.ts.
  • Explain grant validation rules: no slashes at edges, no empty segments, no traversal, no wildcards, no whitespace.
  • Explain child invocation inheritance and narrow-only behavior.
  • Explain resource provider responsibilities: receive grants, inject extra context, register tools, validate operations.
Phase 2

Add schema coverage for resources:

Move the existing resource schema reference from root docs into the site schema section, then link it from common fields, agent kinds, and memrez.

  • Document kind defaulting to the resource name.
  • Use mode: read and mode: read-write. Do not use stale "read-only" wording.
  • Clarify that namespace is provider config, not an automatic runtime grant.
  • Show generated tool names and collision behavior, including hyphen-to-underscore prefixes.
Phase 3

Add memrez as the first resource provider

Create a dedicated page and package README that show the same end-to-end flow in TypeScript and Python: install, create a memrez instance, register the memory provider, declare a memory resource in YAML, pass a trusted context grant at run time, and read/write memories.

  • Add packages/site/src/components/docs/pages/tools/memory-memrez.ts.
  • Add packages/memrez/README.md for npm users.
  • Update python/README.md parity and examples to include provider + SQLite/Postgres memory stores.
  • Add short examples under examples/agents only if the current example set lacks a memory resource manifest.
Phase 4

Update run surfaces and compatibility docs

Once the concepts page exists, update every API surface that accepts context so users can find the field where they need it.

  • Update /docs/sdk-cli/sdk option table with resources and run examples with context.
  • Update /docs/sdk-cli/client with hosted context examples for TS and Python.
  • Update /docs/deploy/http-api request JSON to include context.
  • Update /docs/compatibility with rows for resources, namespace grants, memory provider, memrez stores, and Python parity.
Phase 5

Make LLM docs complete and hard to stale

Because /llms.txt comes from the same manifest, adding pages to DOCS_GROUPS should be enough. Still verify raw docs, anchors, and copied markdown for each new page.

  • Confirm /llms.txt includes the new context, resources, and memrez pages in navigation order.
  • Confirm each new page works as raw markdown at /docs/<slug>.md.
  • Use explicit first paragraphs and tables; LLMs should not have to infer the difference between context and contextIds.
  • Add a lightweight docs coverage checklist or test that checks required topic strings in DOCS_GROUPS.

id: support-with-memory

model:
  provider: openai
  name: gpt-5.4
instruction: |
  Help the user. Use memory when it is relevant.
resources:
  memory:
    mode: read-write
    autoScan: true

TypeScript embedded setup

import { agntz } from "@agntz/sdk";
import { createMemrez, SqliteMemoryStore } from "@agntz/memrez";

const memrez = createMemrez({
  store: new SqliteMemoryStore("./memory.db"),
});

const client = await agntz({
  agents: "./agents",
  resources: { memory: memrez.provider() },
});

await client.agents.run({
  agentId: "support-with-memory",
  input: "Remember that I prefer metric units.",
  context: [`app/user/${userId}`],
});

Python embedded setup

from agntz import LiteLLMModelProvider, agntz
from agntz.memrez import create_memrez
from agntz.memrez_sqlite import SqliteMemoryStore

memrez = create_memrez(store=SqliteMemoryStore("./memory.db"))

client = agntz(
    agents="./agents",
    resources={"memory": memrez.provider()},
    model_provider=LiteLLMModelProvider(),
)

client.agents.run(
    agent_id="support-with-memory",
    input="Remember that I prefer metric units.",
    context=[f"app/user/{user_id}"],
)

LLM Docs Requirements

Every new page must be imported into DOCS_GROUPS. That one step updates the sidebar, static routes, per-page raw markdown, and /llms.txt.

Writing constraints

  • Open each new page with a direct definition and when-to-use guidance.
  • Use the exact field names context, contextIds, resources, mode, and namespace.
  • Prefer tables for contrasts, especially context vs contextIds and local vs hosted.
  • Keep code snippets complete enough to paste into a repo.

Raw markdown checks

  • /docs/concepts/context-and-resources.md returns the same page content.
  • /docs/schema/resources.md includes all resource field references.
  • /docs/tools/memory-memrez.md includes both TS and Python examples.
  • /llms.txt includes the new pages in a sensible order, not only links to them.

Open Questions

Do we expose TS namespacePolicy through @agntz/sdk?

Core supports namespacePolicy, and Python local agntz() exposes namespace_policy. The TS embedded wrapper currently exposes resources but not namespacePolicy. Docs should either avoid TS SDK examples for protected namespace policy or add the option first.

Where should memrez live in navigation?

It can live under Tools because it creates model-visible tools, or under a new Resources group. The smaller change is Tools now, with a future Resources group if RAG/files become public.

Is root docs/ still published?

If yes, it needs the same updates or a sync process. If no, scripts and maintainer notes should stop implying it is the public docs source.

Should the hosted worker mint grants automatically?

Current docs should say trusted application/server code passes context. If hosted worker policies later mint workspace/user grants from auth, document that as a separate deployment feature.
