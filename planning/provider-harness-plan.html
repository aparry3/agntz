01Motivation

A bug we want to never ship again.

Last week a model's tool calls weren't getting appended to the messages array correctly. The model worked. The SDK didn't crash. A user noticed because their downstream logic broke. By the time we found it, we'd already shipped.

That class of bug — a quiet SDK regression on one provider, one model, one capability — has two properties that make it hard to catch with normal tests:

  1. It only shows up against a real provider. Mocks won't reproduce it.
  2. It only shows up in roundtripped state — the model returned something, the SDK accepted it, the next turn was wrong. End-to-end is the only viewpoint.

The harness is the answer: a small, cheap, deterministic battery of capability tests run against every (provider, model) combination we claim to support. Run it on demand or on a cron. If something breaks, we want to find out before a user does.

The harness exists to catch SDK regressions and quiet model changes — not to grade output quality.
Two failure surfaces Bugs that originate in our SDK code (most actionable) and changes from their side — a model swapping its message shape, dropping a capability, or tightening a content policy.
The recent bug Tool-call entries weren't being appended to messages on a specific model. A structural snapshot of the post-tool-result turn would have caught it instantly. See §07.
02Scope

Functionality, not accuracy.

The harness asks one question per test: did the SDK and the model do the structural thing they were supposed to do? Not how well, not how fast, not how cheaply.

§In scope

  • The SDK accepts a valid request shape and returns a valid response shape.
  • Streaming yields chunks of the expected type and a final usage record.
  • Tool calls roundtrip — call, result, next assistant turn — with messages in the documented structure.
  • Multimodal inputs are accepted by models that claim to support them, and politely rejected by models that don't.
  • Negative paths return typed, catchable errors (not raw HTTP fragments or silent successes).

§Out of scope

  • Output quality, accuracy, factuality, reasoning ability.
  • Latency and throughput benchmarks.
  • Cost optimisation. We confirm presence of cost data; we don't compare values.
  • Provider-specific features outside agntz's surface area (e.g. assistants API, custom embeddings).

The discipline matters because it determines what counts as a regression. "Claude got the answer wrong" is not a harness failure. "Claude returned a malformed tool-call entry" is.

Why not vitest? Vitest is great for unit tests but fights matrix orchestration. We want full control over result bucketing, per-provider concurrency, and the difference between "model said no, as expected" and "SDK crashed." That's a runner we write, not one we configure.

04Triage

Six buckets, one decision.

A pass/fail boolean isn't enough. When a multimodal test "fails" on a text-only model, that's expected and green. When a tool-call test fails because our SDK threw, that's a red alert. The bucketer encodes that nuance.

Pass
All good
Capability claimed, capability delivered, response shape matches expectation.
Expected unsupported
Politely declined
Matrix says model doesn't support this; model said so cleanly. Counts as green.
Unexpected unsupported
Model regressed
Matrix says supported; model returned a "not supported" error. Either the model changed or our matrix is wrong.
SDK error
Our bug
Our SDK threw, swallowed, or produced malformed state. The thing this harness exists to find.
Provider error
Their bug
Provider returned a 5xx, malformed response, or stream that ended uncleanly. Worth retrying once.
Timeout
Inconclusive
Call exceeded the per-test budget. Triage separately — might be a real hang, might be a slow model.

§How the runner decides

# given a test and a (provider, model) if matrix[provider][model][capability] === supported: if request succeeded and shape matches → PASS if SDK threw before the network → SDK_ERROR if SDK threw on the response shape → SDK_ERROR if provider replied "unsupported" → UNEXPECTED_UNSUPPORTED if provider 5xx or stream cut → PROVIDER_ERROR if ran past per-test budget → TIMEOUT else: # capability not claimed by this model if provider replied "unsupported" → EXPECTED_UNSUPPORTED if request succeeded → UNEXPECTED_UNSUPPORTED # matrix outdated — capability added if SDK threw → SDK_ERROR # we should refuse cleanly if provider 5xx → PROVIDER_ERROR

The cases that page a human are SDK_ERROR (always) and UNEXPECTED_UNSUPPORTED (either the matrix is stale or the model changed). Everything else is a yellow or green and can be reviewed at leisure.

Why "expected unsupported" is green A text-only model saying "I can't process images" is the system working as designed. It tells us the SDK successfully sent the request and the model gave us a clean refusal. Both surfaces verified.
SDK vs provider error The seam is whether our code produced or accepted malformed state. A 503 from the provider is theirs. A TypeError from our parser is ours.
Provider errors get a single retry 5xx and stream cuts are usually transient. One retry with backoff before marking PROVIDER_ERROR keeps the report quiet.
05Matrix

The capability matrix is the contract.

For each (provider, model) we declare what the model is supposed to be able to do. That declaration is the harness's source of truth. When reality and declaration diverge, we want to know which one is wrong.

06Runtime

One runner, parallel per provider.

The harness is a plain TypeScript program. No test framework. The entry point reads the matrix, expands it into a flat task list (every test × every applicable model), schedules work with a per-provider concurrency limit, and aggregates results. 

typesinterface TestDefinition {
  id: string;
  capability: Capability;
  timeoutMs?: number;            // default 30s
  run: (model: ProviderModelEntry, ctx: RunCtx) => Promise<TestOutput>;
}

type ResultBucket =
  | 'PASS'
  | 'EXPECTED_UNSUPPORTED'
  | 'UNEXPECTED_UNSUPPORTED'
  | 'SDK_ERROR'
  | 'PROVIDER_ERROR'
  | 'TIMEOUT';

interface TestResult {
  test: string;
  provider: Provider;
  model: string;
  bucket: ResultBucket;
  durationMs: number;
  error?: { name: string; message: string; stack?: string };
  snapshotDiff?: string;          // for structural snapshot tests
}

§Concurrency

Each provider has its own semaphore (default 4). All providers run in parallel; within a provider, at most N concurrent calls. This keeps us under rate limits while still finishing in ~minutes.

Why a plain TS program We need result categorisation, not pass/fail. We need per-provider concurrency. We need to attach structural snapshots to specific cases. Vitest can do bits of this with great effort; a 100-line orchestrator is easier and more honest.
Where reports go reports/{ISO-timestamp}.json and reports/{ISO-timestamp}.md, plus a latest.{json,md} symlink. Directory is gitignored — these are run artifacts, not source.
Tiny prompts, tiny outputs Every test is sized for cents, not dollars. "Reply with the word OK." Multimodal uses a 32×32 pixel image. Tool tests invoke a mock get_weather(city) with one arg. Whole-suite cost target: under $0.50 per run.
07Snapshots

Snapshot the shape, not the words.

The tool-call bug was structural: an entry that was supposed to be on the messages array wasn't. Text varies between calls and models — that's noise. The shape of the post-tool-call state should not vary, and that's what we snapshot.

For each snapshot test, we serialise the SDK's resulting messages array (and any other state we care about) with text content replaced by placeholders. On subsequent runs we diff against the stored snapshot.

Snapshot — anthropic / claude-opus-4-7
{
  "messages": [
    { "role": "user",      "content": "<text>" },
    { "role": "assistant", "content": [
        { "type": "tool_use",
          "id":   "<id>",
          "name": "get_weather",
          "input": { "city": "<string>" } }
    ] },
    { "role": "user",      "content": [
        { "type": "tool_result",
          "tool_use_id": "<id>",
          "content":     "<text>" }
    ] },
    { "role": "assistant", "content": [
        { "type": "text", "text": "<text>" }
    ] }
  ]
}
Snapshot — openai / gpt-5.5
{
  "messages": [
    { "role": "user",      "content": "<text>" },
    { "role": "assistant", "content": null,
      "tool_calls": [
        { "id":   "<id>",
          "type": "function",
          "function": {
            "name":      "get_weather",
            "arguments": "<json-string>" } }
      ] },
    { "role": "tool",
      "tool_call_id": "<id>",
      "content":      "<text>" },
    { "role": "assistant", "content": "<text>" }
  ]
}

Different providers have different message shapes — that's fine. We snapshot per (test, provider, model). A drift is either a real SDK change (worth a code review) or a real provider change (worth a matrix update). Either way, we want to see it.

§Update workflow

  • First run: snapshot doesn't exist → write it, mark PASS.
  • Normal run: diff against stored → equal is PASS, different is SDK_ERROR with the diff attached to the result.
  • Intentional change: pnpm run --update-snapshots regenerates. The git diff is the audit trail.
Why this would have caught the bug The recent bug shipped a messages array missing a tool-call entry on a specific model. A snapshot of "what the array looks like after a tool roundtrip" would have diffed on the first run after the regression. The diff would have pointed straight at the missing entry.
Placeholder substitution rules Recursively replace string values that aren't structural — keep keys, types, array lengths; replace text content, ids, and timestamps with sentinels (<text>, <id>, <timestamp>).
08Routes

OpenRouter as a family, not a list.

OpenRouter exposes hundreds of models. Testing all of them isn't the point — testing OpenRouter's pass-through behaviour across model families is. Pick representatives.

§The 8 routes

  1. anthropic/claude-opus-4-7 — Claude family (released April 2026)
  2. openai/gpt-5.5 — OpenAI family (released April 2026)
  3. google/gemini-3.5-flash — Gemini family (Flash released May 19; Pro coming June)
  4. meta-llama/llama-4-maverick — Llama family flagship (released April 2026)
  5. mistralai/mistral-medium-3.5 — Mistral family flagship (released April 2026)
  6. deepseek/deepseek-v4 — DeepSeek family (preview released April 2026)
  7. qwen/qwen-3.7-max — Qwen family (released May 2026)
  8. meta-llama/llama-4-scout — fast / cheap end (smaller Llama 4 sibling)

If OpenRouter is changing its translation layer or rate-limiting a particular upstream, these eight will surface it. The matrix entries for these are filled in per-route, since capabilities differ across families.

Why route diversity OpenRouter normalises across upstream APIs, but the normalisation isn't uniform. A bug in their Claude translation won't reproduce in their Llama translation. Family coverage gives us the most signal per dollar.

09Delivery

Six phases, each shippable.

Each phase ends in a state we can demonstrate by running the harness. No phase leaves us with code that needs a follow-up before it can be exercised.

Phase01

Skeleton + types + matrix

Bootstrap apps/provider-harness. Define Capability, ProviderModelEntry, TestResult, ResultBucket. Hand-write the starter matrix (one model per provider). Entry point loads the matrix and prints it.

running pnpm --filter provider-harness run prints the matrix.

Phase02

Runner core + first test

Per-provider semaphore, parallel orchestration, bucketing logic. Wire up the single-turn test against the full matrix. Stdout-only report for now.

running the harness exercises single-turn against every configured model in parallel.

Phase03

Report writers

JSON report to reports/{ISO}.json; Markdown matrix to reports/{ISO}.md; latest.{json,md} symlinks. Both gitignored.

every run produces a readable artifact you can open in your editor.

Phase04

Real test coverage

Implement all 16 tests from §03. Build the snapshot infrastructure for tool-roundtrip and parallel-tools tests. Wire each test's capability tag so the bucketer can correctly interpret unsupported responses.

the bug class that prompted this work is now under continuous structural watch.

Phase05

OpenRouter subset

Add the 8 routes from §08 with per-route capability entries. No code change beyond matrix additions.

OpenRouter's pass-through correctness is verified against family-representative models.

Phase06

GitHub issue on regression — optional

--report-github flag. On any SDK_ERROR or UNEXPECTED_UNSUPPORTED, gh issue create with the Markdown report attached and a provider-regression label. Default off; manual runs are silent.

a later GH Actions cron can run the harness with the flag and you subscribe to the label.

Pacing Implementer + spec review per task, no per-task quality review. Quality review at the end of each phase. The phases are small enough that this is the right unit.
After phase 6 Cron triggering, Slack alerts, regression auto-bisection, per-model performance trends — all future work that the current shape doesn't preclude.
10Choices

The small set of locked-in calls.

These are the design decisions that shape the rest of the work. Calling them out so they're easy to revisit if any one of them starts to feel wrong during implementation.

Decision Choice Why
Location apps/provider-harness Not a published package; not a workspace dependency. A standalone runnable app, mirroring apps/web's pattern.
Runner Plain TypeScript Matrix orchestration, custom buckets, structural snapshots — we want explicit control over all three. Vitest fights this; a simple script doesn't.
Trigger Manual only (initially) One entry point, run by hand. Cron and CI integration come later if and when we want them.
Reporting JSON + Markdown to disk Self-contained artifacts. Open in editor, diff with git, attach to issues. GitHub issue creation behind a flag.
Concurrency Per-provider semaphore, default 4 All providers parallel; bounded within a provider to avoid rate limits.
Snapshots Structural, with placeholders Model output isn't deterministic. Our roundtrip shape should be. Snapshot what's stable; substitute what isn't.
Matrix coverage One model per provider to start Cheap, fast, covers all of agntz core's provider code paths. Expand once the harness has shape.
OpenRouter 8 family-representative routes Validates OpenRouter's pass-through across families without testing every route.
Prompts Tiny — single tokens where possible Cost target under $0.50 per full run. Functionality, not language modeling.
11Open

Things still to decide.

None of these block Phase 1. Recording them so they don't get lost between phases.

  1. Per-test budget. Default timeout 30s per call. Long-input and streaming-tools may need more. Decide per test in Phase 4.
  2. Snapshot update policy. Today: --update-snapshots regenerates everything. Future: per-test selective update? Probably YAGNI unless we hit it.
  3. Multiple-models-per-provider. Starter matrix has one. When do we add a second? When the first one passes everything and we want broader coverage. Probably after Phase 5.
  4. API key handling. Read from .env. If a provider's key isn't present, skip that provider entirely (not SDK_ERROR). Display "skipped — no credentials" in the report.
  5. Reasoning model output shape. Reasoning tokens vs thinking content — exact field placement differs by provider. May need a small per-provider assertion helper.
  6. Cron home. When we do add a cron (post-Phase 6): Vercel cron, GitHub Actions schedule, or a remote agent? Punt to that conversation.
The point of writing them down is not to answer them now. It's to make sure they aren't forgotten when their time comes.
