Agntz Security Audit

Source-of-truth audit for external company readiness across SDKs, storage, interfaces, auth, worker trust boundaries, dependency exposure, and operational controls.

Audit date: May 25, 2026 Repository: /Users/aaron/Developer/Agntz/agntz Status: Not ready for third-party company use until P0 items are closed

Executive Summary

The current system has a solid foundation in several places, especially user-scoped stores, hashed API keys, encrypted first-class secrets, and tests around run/trace ownership. However, it is not secure enough for other companies to use yet. The blocking issues are current critical dependency vulnerabilities, broad server-side request forgery exposure, plaintext storage of several credential classes, a high-blast-radius internal worker secret, and missing enterprise tenancy/RBAC controls.

The highest-risk pattern is that user-controlled URLs can cause backend services to connect outward from trusted infrastructure across HTTP tools, MCP discovery, MCP connections, manifest validation, token exchange, and webhooks. This needs a central outbound URL policy plus network egress controls, not one-off checks.

Detailed Findings

CriticalSEC-001: Known Clerk and Next auth/middleware bypass vulnerabilities are installed
Area: Dependencies

Evidence

corepack pnpm audit --prod --audit-level=low reports 41 production advisories, including critical Clerk middleware route-protection bypasses and high Next middleware/proxy bypass issues. Installed packages include @clerk/nextjs@7.1.0, @clerk/shared@4.7.0, and next@15.5.15.

Impact

Authentication middleware is part of the main application security boundary. Known bypasses in this layer can expose protected application routes even if route code is correctly written.

Remediation

  • Upgrade Clerk packages to patched versions at or above the advisory fixes.
  • Upgrade Next to a patched version covering middleware/proxy bypasses and SSRF advisories.
  • Upgrade Hono, @hono/node-server, path-to-regexp, fast-uri, js-cookie, postcss, qs, uuid, and ip-address per audit output.
  • Run full auth route regression tests after upgrading.

Acceptance Criteria

  • corepack pnpm audit --prod --audit-level=high returns no high or critical advisories.
  • Protected route tests prove unauthenticated requests cannot reach app APIs.
CriticalSEC-002: SSRF across HTTP tools, MCP, validation, token exchange, and webhooks
Area: Interfaces

Evidence

User-controlled URLs are fetched by backend code in packages/core/src/http-tool.ts, packages/manifest/src/validate.ts, packages/core/src/auth/token-resolver.ts, packages/app/src/app/api/mcp-tools/route.ts, packages/app/src/lib/connections.ts, packages/core/src/mcp/list-tools.ts, packages/core/src/mcp/client-manager.ts, and packages/core/src/webhooks/dispatcher.ts. Most paths only validate that a string parses as a URL, with no private IP, localhost, DNS rebinding, redirect, or allowlist policy.

Impact

An authenticated user, compromised API key, or malicious manifest could use trusted infrastructure to reach internal metadata services, private network endpoints, localhost services, cloud control planes, or internal admin surfaces.

Remediation

  • Implement one shared outbound URL policy used by HTTP tools, MCP, manifest validation, token exchange, webhooks, and image fetches.
  • Default to https: for user-configured outbound integrations unless an explicit admin allowlist permits http:.
  • Resolve DNS before connecting and block loopback, link-local, private, carrier-grade NAT, multicast, unspecified, and reserved ranges for IPv4 and IPv6.
  • Disable redirects or re-run policy checks after every redirect.
  • Add network egress firewalling so application code cannot reach cloud metadata or private network ranges even if a code check is missed.

Acceptance Criteria

  • Regression tests cover localhost, 127.0.0.1, ::1, private IPv4 ranges, private IPv6 ranges, DNS names resolving to private IPs, and redirect-to-private cases for each outbound interface.
  • Allowed public URLs still work for expected integrations.
HighSEC-003: Provider API keys and MCP auth headers are stored in plaintext
Area: Storage

Evidence

ProviderStore writes ar_providers.api_key as plaintext in Postgres and stores full provider config in JSON/memory stores. MCP connection config is stored as JSON and can include authorization headers. The UI states that MCP header values are stored encrypted at rest, but the connection store writes config directly.

Impact

Database read access, backups, logs, local JSON files, or support dumps can expose model provider keys and third-party MCP credentials. This undermines the stronger encrypted SecretStore design.

Remediation

  • Move provider API keys and MCP sensitive headers into SecretStore or encrypted columns using the same or stronger envelope encryption pattern.
  • Store only references and metadata in provider/connection records.
  • Add migrations to encrypt existing values and scrub plaintext from JSON stores and backups where possible.
  • Correct UI copy only after storage actually encrypts the values.

Acceptance Criteria

  • Direct database rows and JSON files do not contain raw provider keys or auth headers.
  • Tests fail if provider or connection credentials are persisted in plaintext.
HighSEC-004: Static internal worker secret can impersonate any user if leaked
Area: Auth

Evidence

packages/worker/src/middleware/auth.ts accepts X-Internal-Secret plus a caller-supplied userId in the JSON body or X-User-Id header. App helper code derives this from Clerk, but the worker itself trusts the internal secret. Deployment docs expose the worker as a normal service URL protected mainly by this shared secret and API keys.

Impact

If WORKER_INTERNAL_SECRET leaks, an attacker can act as any user against internal worker paths that accept the header. The blast radius is all tenants/users the worker can access.

Remediation

  • Place app-to-worker traffic on private networking or mTLS.
  • Replace static shared-secret impersonation with short-lived signed service JWTs containing issuer, audience, expiration, subject, and authorized action.
  • Reject user identity supplied from untrusted request bodies unless bound to a verified app-issued token.
  • Use separate secrets or keys for public API, internal app calls, and admin/system routes.

Acceptance Criteria

  • A leaked external API key cannot call internal routes.
  • A leaked old service token expires quickly and cannot be replayed outside its audience/action.
  • Worker ingress is not publicly reachable for internal-only routes.
HighSEC-005: No organization tenancy, RBAC, or scoped API keys for company use
Area: Tenancy

Evidence

Application context is scoped by Clerk userId through requireUserContext() and adminStore.forUser(userId). API keys resolve to a user owner. Docs note that organizations and API-key scopes are future work.

Impact

Companies need organization tenancy, member roles, least-privilege keys, and auditability. User-only ownership is not enough for multi-person company accounts or enterprise security reviews.

Remediation

  • Add an explicit tenant model such as orgId plus userId, with all stores and indexes scoped by tenant.
  • Add roles and permissions for owner, admin, developer, viewer, and service account use cases.
  • Add API key scopes, expiration, last-used metadata, rotation, and per-key revocation.
  • Add tenant-level audit logs for secrets, keys, providers, connections, runs, and admin changes.

Acceptance Criteria

  • Tests prove one organization cannot read or mutate another organization's agents, runs, traces, secrets, providers, keys, or connections.
  • Least-privilege API keys cannot perform actions outside their scope.
HighSEC-006: Secret encryption key is required by code but omitted from deployment docs
Area: Operations

Evidence

packages/core/src/utils/crypto.ts requires a 32-byte AGNTZ_SECRET_KEY for AES-256-GCM secret encryption/decryption. Deployment docs describe WORKER_INTERNAL_SECRET and many app variables, but omit AGNTZ_SECRET_KEY. The inspected local env did not define it.

Impact

Deployments can silently ship without a required encryption key until secret operations fail, or teams may improvise unsafe key handling. There is also no documented rotation/key-version workflow.

Remediation

  • Add AGNTZ_SECRET_KEY generation, storage, backup, and rotation instructions to deployment docs and examples.
  • Prefer KMS/envelope encryption for production rather than a raw app env key.
  • Add key versioning so old secrets can be decrypted during rotation and re-encrypted safely.
  • Fail startup if secret-backed features are enabled without a valid key.

Acceptance Criteria

  • Fresh deployment docs produce a working encrypted secret store.
  • A key rotation test decrypts old values and re-encrypts them under the new key.
HighSEC-007: Webhook signatures lack timestamp/replay protection and callback URL policy
Area: Webhooks

Evidence

packages/core/src/webhooks/dispatcher.ts signs only the JSON body using HMAC and sends a caller-provided callbackUrl. The run route checks that callback URL and secret name are both present, but does not enforce an outbound URL policy. The signature does not include timestamp, delivery ID, or replay window.

Impact

Captured webhook bodies and signatures may be replayable. Callback URLs can also be used as another SSRF path from the worker.

Remediation

  • Apply the shared outbound URL policy to webhook callback URLs.
  • Add a timestamp header and sign timestamp.deliveryId.body.
  • Document receiver verification with a short tolerance window and replay cache.
  • Expose secret rotation and dual-signing or grace-period verification for webhook secrets.

Acceptance Criteria

  • Old webhook signatures are rejected outside the tolerance window.
  • Callback URLs to private/localhost destinations are blocked.
MediumSEC-008: Worker CORS is unrestricted
Area: Worker API

Evidence

packages/worker/src/routes.ts applies app.use("*", cors()) globally. The worker uses bearer API keys and internal headers rather than cookies, so this is not classic CSRF, but it broadens browser-origin exposure.

Impact

If users paste API keys into browsers or a future UI stores credentials in browser context, arbitrary origins can make worker calls. It also makes it harder to reason about intended clients.

Remediation

  • Restrict CORS to configured app origins and documented SDK browser origins, or disable CORS for non-browser APIs.
  • Do not allow internal routes from browser origins.

Acceptance Criteria

  • Unexpected origins fail preflight and cannot call worker endpoints from a browser.
MediumSEC-009: Request body and resource limits are incomplete
Area: Availability

Evidence

Worker routes parse JSON bodies for runs, validation, and configuration without a central request-size limit. /build-agent caps some fields and the image fetcher caps fetched image size, but the API surface lacks a consistent body, stream, timeout, and concurrency policy. The audit also flagged Hono body-limit related advisories in the installed version.

Impact

Large bodies, oversized manifests, long tool sessions, or repeated expensive calls can consume memory, worker slots, and model spend.

Remediation

  • Add edge and framework-level body limits for app and worker routes.
  • Add per-user and per-key rate limits for run creation, validation, MCP discovery, build-agent, and webhook delivery.
  • Set maximum manifest size, step count, input size, output size, and run duration.

Acceptance Criteria

  • Oversized requests fail before full body parsing.
  • Rate-limit tests cover authenticated, API-key, and unauthenticated expensive endpoints.
MediumSEC-010: Logs can include user prompts, tool params, and sensitive snippets
Area: Privacy

Evidence

packages/core/src/model-provider.ts logs model error context with message previews and content snippets. packages/worker/src/bridge.ts logs serialized tool params truncated to 200 characters. Hosted tracing defaults recordIO to false, which is good, but SDK/local tracing can record IO when enabled.

Impact

User prompts, tool arguments, provider responses, or pasted credentials can appear in operational logs. Company customers often treat prompt and run data as confidential.

Remediation

  • Disable prompt and tool-argument previews in production logs by default.
  • Use structured redaction for known secret patterns, API keys, authorization headers, and configured secret values.
  • Document trace retention, export, deletion, and IO-recording controls.

Acceptance Criteria

  • Tests prove common secret patterns and configured secret values do not appear in logs or traces.
MediumSEC-011: Image fetcher SSRF guard misses DNS/private-resolution checks
Area: Interfaces

Evidence

packages/core/src/image-fetcher.ts blocks literal localhost/private IPv4/IPv6 inputs and enforces content type, timeout, and max bytes. It does not resolve hostnames before connecting, so a DNS name that resolves to a private address is not blocked.

Impact

A malicious image URL can bypass literal IP checks through DNS and reach private addresses if network egress permits it.

Remediation

  • Move image fetching onto the same shared outbound URL policy described in SEC-002.
  • Resolve and validate all IPs before connect, and re-check redirects.

Acceptance Criteria

  • Hostnames resolving to private or loopback IPs are rejected in tests.
MediumSEC-012: Bundled read-file tool uses a weak path prefix boundary check
Area: Local Tools

Evidence

packages/worker/src/tools/read-file.ts resolves paths under bundled agent-builder references and checks target.startsWith(REFS_DIR). Prefix checks are weaker than path-relative boundary checks when sibling paths share a prefix.

Impact

Current blast radius appears limited to a bundled system-agent helper, but this pattern can become dangerous if reused for user-controlled local file access.

Remediation

  • Replace the prefix check with path.relative(REFS_DIR, target) and reject paths starting with .. or absolute paths.
  • Add traversal tests for sibling-prefix cases.

Acceptance Criteria

  • Traversal and sibling-prefix test cases fail closed.
LowSEC-013: Local env file contains credential hygiene issues
Area: Operations

Evidence

The parent local env file contained a commented database connection string and did not define AGNTZ_SECRET_KEY. The value is not repeated here.

Impact

Commented secrets are still secrets. They can be copied, indexed, committed, or exposed in screenshots and support bundles. Missing local encryption config also masks deployment-readiness problems.

Remediation

  • Rotate any credential that appeared in local env comments.
  • Remove commented secret values from env files.
  • Add secret scanning to pre-commit and CI.

Acceptance Criteria

  • Secret scan passes on the repository and local env templates contain placeholders only.
LowSEC-014: SDK docs should explicitly warn against browser API-key use
Area: SDKs

Evidence

The TypeScript client accepts bearer API keys for worker API access. The worker CORS policy is broad. There should be clear docs and types guiding company customers to use API keys from server-side environments only unless a future scoped browser token model is created.

Impact

Customers can accidentally embed long-lived API keys in browser apps, exposing account-level access.

Remediation

  • Document server-only usage for API keys in TS and Python SDK docs.
  • Consider naming or runtime warnings similar to dangerouslyAllowBrowser if browser use is ever supported.
  • Prefer short-lived scoped browser tokens for future browser workflows.

Acceptance Criteria

  • SDK docs and examples do not encourage exposing long-lived API keys to browsers.
LowSEC-015: Python package lacks a lock/audit workflow in this checkout
Area: Supply Chain

Evidence

python/pyproject.toml uses lower-bound dependency ranges such as httpx>=0.28.0, pydantic>=2.8.0, PyYAML>=6.0.0, and optional litellm>=1.75.0. No Python lockfile or local audit tool was available in this checkout. Local Python was 3.9 while the package requires 3.11 or newer.

Impact

SDK releases can resolve to different dependency versions across environments, and Python dependency advisories are not currently part of the observed audit loop.

Remediation

  • Add a Python dependency lock and release/audit workflow using Python 3.11 or newer.
  • Run pip-audit or an equivalent in CI for the Python package.

Acceptance Criteria

  • Python package CI runs install, tests, and dependency audit on supported Python versions.

What Looks Solid

PositivePer-user store scoping

App context derives user identity from Clerk and wraps stores through forUser(userId). Tests cover run, trace, and store isolation across memory, SQLite, and Postgres paths.

PositiveFirst-class SecretStore encryption

Secrets use AES-256-GCM with a required 32-byte key. App APIs return metadata rather than raw decrypted values.

PositiveAPI key entropy and hashing

API keys are generated with high entropy, stored as SHA-256 hashes, shown once, and support revocation.

PositiveHosted tracing defaults

Worker-created span emitters set recordIO: false by default, reducing accidental prompt/output capture in hosted traces.

PositiveToken cache owner isolation

HTTP auth token resolver cache keys include owner context, reducing cross-user token reuse risk.

PositiveWebhook signing foundation

Webhooks already use HMAC signatures and delivery IDs; this foundation needs timestamp/replay hardening rather than a full redesign.

Prioritized Remediation Plan

P0: Block External Company Use Until Complete

  1. Patch critical/high dependency advisories for Clerk, Next, Hono, and transitive runtime packages.
  2. Build and enforce a shared outbound URL policy across HTTP tools, MCP, validation, token exchange, image fetches, and webhooks, backed by network egress controls.
  3. Encrypt provider API keys and MCP auth headers at rest; migrate existing plaintext values.
  4. Replace public static internal worker-secret impersonation with private networking and short-lived service authentication.
  5. Add AGNTZ_SECRET_KEY deployment guidance, startup validation, and a rotation path. Rotate the local credential that appeared in an env comment.

P1: Company Readiness

  1. Add organization tenancy, RBAC, scoped API keys, key expiration, and tenant-level audit logs.
  2. Add request body limits, run limits, rate limits, and cost controls for expensive routes.
  3. Harden webhook signatures with timestamp, delivery ID binding, receiver docs, and replay protection.
  4. Restrict worker CORS and document SDK server-only API-key handling.
  5. Remove sensitive prompt/tool previews from production logs and add redaction tests.

P2: Assurance And Compliance

  1. Add CI dependency audit gates for TypeScript and Python packages.
  2. Add SOC 2 readiness controls: access review, retention/deletion, audit export, incident response, backup restore tests, and change-management evidence.
  3. Add security regression tests for all fixed findings and keep this report updated as items close.
